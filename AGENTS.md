## 🧭 行为准则
- 中文不可违：自然语言说明使用中文；代码、命令、路径、报错保留原文，必要时附中文解释。
- 思考先于操作：先分析需求、约束、依赖，再动手。
- 任务拆解：运用金字塔结构自顶向下拆分，逐步交付可验证结果。
- 主动澄清：高风险、接口契约不明、多方案成本差异大时必须提问；其余情况可先基于合理假设执行，并在交付时说明假设。
- 记忆闭环：先读与任务相关的变更记录、README 和关键文档，再做当前任务；交付前回看是否需要补充项目记忆，默认宁记勿漏。
- 失败复盘：同一问题连续失败 3 次需停下，总结原因与新策略。
- 反馈风格：简洁直接，不奉承；如需指出问题可幽默提示但不可冒犯。

### 🛠️ 实施阶段
- [ ] 设计实现方案与验证手段，再逐步编码。

### ✅ 收尾阶段
- [ ] 运行必要的测试或脚本；若受限需说明原因与补救建议。
- [ ] 使用 `git status` 确认改动范围，准备交付说明。
- [ ] 若存在项目需求变更、关键决策或较大代码变更，补充记录到当前项目实际使用的变更文件，当前默认是根目录 `CHANGELOG.md`。
- [ ] 更新变更记录时，同步维护文件头部或条目中的更新时间字段。
- [ ] 记录内容至少包含：修改时间、变更背景、核心改动、影响范围、兼容性或风险、验证情况；若未验证需说明原因。

## 🧠 记忆机制
- 记忆载体：当前仓库默认以根目录 `CHANGELOG.md` 作为项目级、可追溯的事实记忆；如未来新增 `memory/CHANGELOG.md`，可按需要切换到更明确的统一载体。
- 读取顺序：每次任务启动时，先读取与当前任务直接相关的最近记录，再结合仓库代码、配置与文档继续执行。
- 项目事实优先：涉及需求变更、架构决策、接口契约、配置策略、联调口径、页面行为、接口字段或部署约定等项目事实时，必须回写到正式变更记录，禁止只留在临时对话中；纯 `AGENTS.md`、输出风格或协作提示词调整默认不记录。
- 读取原则：只关注与当前任务直接相关的记忆；若变更记录与代码、配置存在冲突，以仓库内事实为准，并在交付时说明差异。
- 触发条件：满足以下任一情况必须记录。
  - 用户调整需求范围、优先级、验收标准、接口约定、配置开关或运行策略。
  - 修改核心业务流程、跨页面共享逻辑、公共组件、共享数据结构、后端 API 契约、上传与登录策略等关键内容。
  - 改动涉及多个模块，或足以影响后续任务理解、排障、联调和回滚判断。
- 记录原则：优先采用 `Keep a Changelog` 风格，归类到 `Added`、`Changed`、`Fixed`、`Removed`；描述要简洁、具体、可检索。
- 时间要求：变更记录需维护明确的修改时间；新增或更新重要记录时，条目内也应显式写出“修改时间”。
- 记录时机：默认在任务收尾时补充；若任务跨度大或中途发生关键需求转向，可先追加草稿，交付前整理完成。
- 决策准则：若不确定是否算“较大变更”，按需要记录处理，避免遗漏。

## 🧩 项目适配约束
- 当前项目以 `uni-app` + `Vue 3` + `uview-plus` 为前端主栈，以微信小程序为主要运行目标；改动前优先确认页面、路由、平台 API 和样式约束是否匹配。
- 当前仓库同时包含 `backend/` Go 服务端代码；涉及前后端联动时，需同时校验接口契约、登录态、上传链路与配置项是否一致。
- 优先遵守仓库已有目录与命名方式，如 `pages/`、`utils/`、`docs/`、`backend/internal/` 等，不强行套用其他项目的 DDD、ES 或索引约定。
- 若任务涉及微信登录、分享邀请、图片上传或 HBuilderX/微信开发者工具编译链路，应优先查阅已有 README 与 `docs/` 文档，再决定实现方式。

## ✨ 终端输出风格
- 语气：专业友好，直击重点，可适度使用 🎯/✨/💡/⚠️ 等 Emoji。
- 结构：标题与列表层次分明，行宽尽量控制在 80 字符以内，充分留白。
- 内容：先给结论后给细节，必要时用代码块并附语言标签；多项对比、方案权衡或结构化信息并列展示时优先使用 Markdown 表格。
- 引用：路径使用反引号（如 `utils/app-config.js`），引用单行行号格式 `file:42`。
- 交互：长流程提供阶段性反馈，错误要说明原因和可行补救措施。
