# 美食库模块 UI 设计优化评估文档

修改时间：2026-04-29 CST
适用范围：`pages/index/index.vue`、`pages/index/components/library-header-section.*`、`pages/index/components/recipe-card-item.*`、`pages/index/recipe-card.js`
参考来源：`我们的美食空间/` 原型目录、`ui-ux-pro-max` 本地设计库
目标平台：微信小程序，`uni-app + Vue 3 + uview-plus`

## 1. 结论

当前小程序的 `美食库` 已经完成了一部分原型风格迁移：页面使用暖白底、卡片化头部、菜单 spotlight、搜索筛选面板、轻浮层底部导航和图像型菜谱卡，整体方向与 `我们的美食空间/` 中的共享数字厨房风格一致。

本轮评估建议继续做“美食库模块整体 UI 收口”，重点不是替换业务链路，而是让美食库在视觉层级、空态动作、筛选反馈和菜卡可读性上与已经改造过的 `空间` 模块保持同一套设计语言。

推荐优先落地 4 类改动：

1. 搜索与筛选面板组件化，并给 `全部 / 想吃 / 吃过` 增加数量反馈。
2. 空态从“说明文案”升级为“可继续操作的结果页”，补 `清除搜索`、`添加这道菜`、`查看全部` 等动作。
3. 菜谱卡片继续保留当前状态 switch，但强化触控面积、状态语义和摘要兜底。
4. 菜单 spotlight 更靠近原型 `PlanCard` 的日期卡表达，形成“美食库 -> 安排菜单 -> 菜谱列表”的连续工作流。

本次文档仅做 UI 方案与改动评估，不修改运行时代码、不改变后端接口、不改变点菜模式、邀请链路或菜谱保存契约。

### 2.2 `ui-ux-pro-max` 设计建议吸收

| 设计库建议 | 适配结论 |
| --- | --- |
| `Marketplace / Directory`：搜索是主 CTA，推荐词降低搜索成本 | 美食库本质是私有菜谱目录，搜索框应保留首屏核心位置 |
| `Nature Distilled`：土色、陶土、橄榄绿、柔和阴影 | 比 `Restaurant/Food Service` 的高饱和红金更适合家庭共享厨房 |
| `Bento Grids`：模块化卡片、清晰层级、轻圆角 | 适合菜单卡、筛选卡、空态卡，但列表密度不能过低 |
| `Glassmorphism`：毛玻璃适合导航和浮层 | 仅用于底部导航、邀请主按钮等少数浮层，避免正文卡片对比不足 |
| 移动端触控：按钮建议接近 `44px` 触控目标，邻近目标保持间距 | 主操作建议 `80rpx - 96rpx`，筛选与 switch 至少保留当前 `56rpx - 68rpx` 可点高度 |
| 搜索无结果：不要只显示 `0 results`，应给建议和下一步 | 当前空态需要补动作和推荐关键词 |

## 4. 建议设计系统

### 4.1 色彩

不建议照搬 `ui-ux-pro-max` 食品类高饱和红色主色，因为当前产品更偏家庭共享、温暖记录和轻决策。建议延续已有暖色体系：

| 角色 | 建议色值 | 用途 |
| --- | --- | --- |
| 页面背景 | `#F6F4F1` / `#F8F5EF` | 全局背景 |
| 卡片背景 | `#FFFDF8` | 菜卡、搜索面板、空态 |
| 主文本 | `#2F2923` | 标题、菜名 |
| 次文本 | `#74685E` | 描述、说明 |
| 主品牌棕 | `#5C4033` | 主操作、选中态、底部 FAB |
| 陶土橙 | `#BF715F` | `想吃`、安排菜单、提醒 |
| 鼠尾草绿 | `#6F846A` | `吃过`、成功、已完成 |
| 柔和边框 | `rgba(91, 74, 59, 0.08)` | 卡片边界 |
| 柔和阴影 | `0 12rpx 24rpx rgba(56,44,30,0.05)` | 列表卡片 |

### 4.2 圆角与层级

| 元素 | 建议 |
| --- | --- |
| 搜索筛选面板 | `24rpx - 28rpx`，保持比空间大卡更轻 |
| 菜谱卡 | `24rpx - 26rpx`，当前基本合适 |
| 封面图 | `18rpx - 20rpx`，当前基本合适 |
| 状态筛选 | `18rpx - 20rpx`，确保文字不拥挤 |
| 空态卡 | `26rpx - 30rpx`，承载主/次动作 |
| 底部导航 | 保持现有浮动胶囊，不纳入本轮变更 |

### 4.3 字体

- 正文继续使用系统无衬线，保证微信小程序端稳定。
- `美食库` 标题可继续无衬线，避免与 `空间` 名称的衬线气质混淆。
- 菜单 spotlight 中的日期或数量可尝试系统衬线数字栈，强化“菜单卡”记忆点。
- 不引入 Google Fonts 或远程字体。

## 5. 目标界面结构

```text
美食库
├─ 标题区
│  ├─ 图标 + 美食库
│  ├─ 按餐别整理，想吃和吃过更清楚
│  └─ 安排菜单
├─ 菜单 spotlight（仅存在记录时展示）
│  ├─ 日期 / 状态
│  ├─ 菜品摘要
│  └─ 进度或进入符号
├─ 搜索筛选面板
│  ├─ 搜索框
│  ├─ 最近搜索 / 推荐词
│  ├─ 早餐 / 正餐 + 数量
│  └─ 全部 / 想吃 / 吃过 + 数量
├─ 列表标题行
│  ├─ 当前筛选摘要
│  ├─ 清除筛选
│  └─ 帮我选
├─ 菜谱卡列表
└─ 空态 / 搜索无结果态
   ├─ 结果说明
   ├─ 主动作
   └─ 次动作
```

## 6. 建议改动清单

### P0-1：搜索筛选面板组件化

建议文件：

- 新增 `pages/index/components/library-toolbar-section.vue`
- 新增 `pages/index/components/library-toolbar-section.scss`
- 调整 `pages/index/index.vue`

核心改动：

- 将搜索框、最近搜索/推荐词、餐别 tab、状态 tab 从 `index.vue` 移入组件。
- `index.vue` 保留业务状态和方法，通过 props / emits 传入。
- 给状态 tab 增加数量：
  - `全部`：当前餐别下全部数量
  - `想吃`：当前餐别下 `wishlist` 数量
  - `吃过`：当前餐别下 `done` 数量

影响评估：

| 维度 | 评估 |
| --- | --- |
| 改动程度 | 中 |
| 业务风险 | 低，事件名保持即可 |
| 主要收益 | 降低 `index.vue` 模板体积，让美食库 UI 可单独迭代 |
| 注意事项 | 点菜模式下仍只保留搜索，不展示普通筛选 |

### P0-2：空态补动作

建议文件：

- 新增 `pages/index/components/library-empty-state.vue`
- 新增 `pages/index/components/library-empty-state.scss`
- 调整 `pages/index/index.vue`

建议状态：

| 场景 | 标题 | 主动作 | 次动作 |
| --- | --- | --- | --- |
| 搜索无结果 | `没有找到“xxx”` | `清除搜索` | `添加这道菜` |
| 当前餐别为空 | `还没有早餐记录` | `添加早餐` | `切到正餐看看` |
| 当前状态为空 | `正餐里还没有想吃的菜` | `查看全部` | `添加一道正餐` |
| 同步失败且无缓存 | `暂时无法连接空间` | `重试同步` | `去空间页查看` |

影响评估：

| 维度 | 评估 |
| --- | --- |
| 改动程度 | 中 |
| 业务风险 | 中，需要新增空态动作处理函数 |
| 主要收益 | 搜索无结果和空库不再是死路 |
| 注意事项 | `添加这道菜` 若要预填搜索词，需要扩展 `openAddSheet` 或新增包装方法 |

### P0-3：菜谱卡可读性与触控增强

建议文件：

- `pages/index/components/recipe-card-item.vue`
- `pages/index/components/recipe-card-item.scss`
- `pages/index/recipe-card.js`

核心改动：

- 保留当前状态 switch，不改成原型里的两个独立小按钮，避免破坏现有快捷切换习惯。
- switch 外层触控面积保持或略增，目标不低于 `64rpx` 高；如果视觉高度不变，可通过外层 padding 扩大实际点击区。
- `buildRecipeListSummary` 从只读 `summary` 改为兜底读取 `ingredient`、`note` 或解析出的前 1 条步骤，减少菜卡正文空白。
- 无图菜卡继续保持与有图菜卡接近高度，避免列表跳动。

影响评估：

| 维度 | 评估 |
| --- | --- |
| 改动程度 | 小到中 |
| 业务风险 | 低 |
| 主要收益 | 菜卡信息更稳定，状态切换更好点 |
| 注意事项 | `@tap.stop` 必须保留，防止状态切换触发详情页 |

### P0-4：菜单 spotlight 升级为计划卡

建议文件：

- `pages/index/components/library-header-section.vue`
- `pages/index/components/library-header-section.scss`

核心改动：

- 当前无菜单记录时继续隐藏，不恢复空态菜单卡，保持 2026-04-25 既有决策。
- 有菜单记录时，视觉上更接近原型 `PlanCard`：
  - 左侧使用日历图标和日期。
  - 中间展示菜品摘要。
  - 右侧展示 `1/3` 或箭头。
  - 背景用 `#FFFDF8 -> #F0EDE6` 轻渐变，弱陶土橙点缀。
- 多菜单记录的左右滑动和点击进入详情保持不变。

影响评估：

| 维度 | 评估 |
| --- | --- |
| 改动程度 | 小到中 |
| 业务风险 | 低 |
| 主要收益 | 强化“安排菜单”与“美食库”的关系 |
| 注意事项 | 不新增无记录 spotlight，避免占首屏空间 |

## 7. P1 后续优化

| 优化项 | 说明 | 建议时机 |
| --- | --- | --- |
| 搜索无结果预填新增 | 点击 `添加这道菜` 时，把搜索词预填到 `draft.title` | P0 空态稳定后 |
| 顶部同步状态轻提示 | 在标题区辅助说明中区分 `同步中 / 同步失败 / 已连接空间` | 与数据加载体验一起做 |
| 筛选结果保存 | 保留用户上次餐别或状态筛选 | 需要确认产品预期后再做 |
| AI 助手入口 | 原型中空态和底部 FAB 偏 AI；当前不建议直接照搬 | 等 AI sheet 需求明确后 |
| 标签/收藏入口 | 原型菜卡偏轻，但后续可扩展标签和收藏 | 菜谱管理能力升级时 |

## 8. 不建议本轮做的事

| 不建议项 | 原因 |
| --- | --- |
| 直接迁移 React/Tailwind/Lucide 依赖 | 小程序技术栈不同，迁移成本高且无必要 |
| 把底部中间按钮立即改为 AI 助手 | 当前真实链路是添加菜谱，贸然改变会影响验收 |
| 大量使用毛玻璃正文卡片 | 低端安卓微信上对比和性能都有风险 |
| 使用高饱和红色作为主色 | 与当前家庭共享厨房气质冲突，也会破坏空间页一致性 |
| 为 UI 重构新增后端接口 | 本轮信息都可由现有 `recipes`、筛选状态和菜单记录计算 |

## 9. 验收清单

### 视觉

- [ ] 美食库首屏能看到标题、安排菜单、搜索框、筛选入口和至少一张菜卡。
- [ ] 搜索筛选面板与空间页卡片风格一致，但密度不低于当前版本。
- [ ] `全部 / 想吃 / 吃过` 数量清楚可见。
- [ ] 空态卡片有主动作和次动作，不只有说明文案。
- [ ] 颜色不只依赖棕色，完成态绿、提醒态陶土橙、中性色分工明确。

### 交互

- [ ] 搜索输入、清除、最近搜索、推荐关键词都保持可用。
- [ ] 切换餐别、切换状态、清除筛选、帮我选逻辑不变。
- [ ] 菜卡主体点击进入详情，状态 switch 不触发详情。
- [ ] 点菜模式下加入菜单按钮不触发详情。
- [ ] 空态动作能正确执行：清除、查看全部、切换餐别、新增菜谱。

### 工程

- [ ] `index.vue` 中美食库 UI 模板明显减少，业务状态仍集中在父组件。
- [ ] 新增组件只使用 props / emits，不直接调用 API。
- [ ] 不改变后端接口、路由、缓存、图片上传、菜谱保存契约。
- [ ] `git diff --check` 通过。
- [ ] 微信开发者工具或真机补测默认列表、搜索无结果、空餐别、状态空、点菜模式。

## 10. 风险与处理

| 风险 | 说明 | 处理 |
| --- | --- | --- |
| 组件拆分影响事件链路 | 搜索、筛选、清除、最近搜索都在 `index.vue` 内 | 拆分时只移动模板和样式，事件仍由父组件实现 |
| 点菜模式被误伤 | 美食库模板有 `isLibraryMealOrderMode` 条件分支 | 每个新增组件都显式传入模式状态，保留原分支 |
| 空态动作引发新增流程变化 | `添加这道菜` 可能需要预填菜名 | 先实现打开新增弹层；预填作为 P1 小步扩展 |
| 触控目标不足 | 状态 switch、清除按钮、筛选 chip 较紧凑 | 通过外层容器扩大实际可点区域，视觉尺寸可保持克制 |
| 视觉过度装饰 | 原型有动效和玻璃质感 | 正文卡片以轻阴影和实色为主，毛玻璃只留给底部导航等浮层 |

## 11. 建议实施顺序

1. 先拆 `library-toolbar-section`，同步补状态数量。
2. 再做 `library-empty-state`，补动作与事件。
3. 调整 `recipe-card-item` 和 `recipe-card.js`，增强摘要兜底和触控。
4. 最后微调 `library-header-section` 的菜单 spotlight。

这个顺序能先降低 `index.vue` 的 UI 复杂度，再补体验断点，最后做视觉 polish，回归风险相对可控。
