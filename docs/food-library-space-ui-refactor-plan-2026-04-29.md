# 美食库与空间 UI 界面重构方案

修改时间：2026-04-29 CST
适用范围：`pages/index/index.vue`、`pages/index/components/*`、`我们的美食空间/` 原型目录
目标平台：微信小程序，`uni-app + Vue 3 + uview-plus`

## 1. 背景与结论

`我们的美食空间/` 是一套独立的 Vite + React 视觉原型，核心定位是“情侣与家庭共享的数字厨房：吃什么决策、菜谱管理与共同饮食安排”。原型在视觉上更强调温暖、轻盈、卡片化、底部浮动导航和 AI 辅助；当前小程序实现已经具备核心业务链路，包括美食库筛选、搜索、随机推荐、点菜模式、空间切换、成员列表、邀请与资料编辑。

本次重构不建议推翻现有页面结构，而应采用“保留业务链路，重整视觉层级和组件边界”的方式：

- `美食库`：强化“找菜、筛菜、选菜、安排菜单”的连续工作流，减少控件拥挤感，补齐搜索无结果和空库时的可执行动作。
- `空间`：从“设置页式信息列表”升级为“当前共享空间仪表盘”，让空间名、成员关系、邀请、切换空间和个人资料更容易理解。
- `底部导航`：保留 `美食库 / 添加 / 空间` 三入口，但视觉上向原型的浮动胶囊导航靠拢，避免当前大面积白色固定栏压缩内容。
- `视觉系统`：沿用当前暖色厨房感，但要加入鼠尾草绿、陶土橙、米白和深墨色形成层级，避免页面只剩棕色系。

## 3. 设计目标

### 3.1 用户目标

- 进入美食库后，能快速回答“今天/周末吃什么”。
- 搜索、筛选、随机推荐和安排菜单之间不互相打架。
- 进入空间页后，能立刻知道“我在哪个空间、有哪些人、能邀请谁、怎么切换空间”。
- 对新用户或空数据用户，要有下一步动作，而不是只看到空说明。

### 3.2 产品目标

- 让原型中的温暖视觉语言落到小程序真实组件。
- 不破坏现有后端接口、点菜模式、缓存、邀请和资料编辑链路。
- 先完成 P0/P1 的 UI 重构，再评估 AI 助手、活动流、成员管理等扩展能力。

### 3.3 技术目标

- 页面仍以 `pages/index/index.vue` 为主承载，避免新增路由打断历史链路。
- 将过重的模板与样式继续下沉到组件，减少 `index.vue` 的视觉代码体积。
- 所有可点击区域满足小程序触控可用性，主操作建议不小于 `64rpx` 高，图标按钮建议不小于 `56rpx`。
- 动画控制在 `180ms - 260ms`，避免列表大量重排时卡顿。

## 4. 设计系统建议

### 4.1 色彩

当前原型和实现都偏暖棕，但后续应建立更明确的 token，避免每个组件手写相近色值。

| 角色 | 建议色值 | 用途 |
| --- | --- | --- |
| 页面背景 | `#F8F5EF` | 全局背景，比纯灰更有厨房温度 |
| 卡片背景 | `#FFFDF8` | 菜卡、空间卡、工具面板 |
| 主文本 | `#2F2923` | 标题、主信息 |
| 次文本 | `#74685E` | 描述、辅助信息 |
| 主品牌棕 | `#5B4638` | 主按钮、选中态、底部导航 |
| 鼠尾草绿 | `#6F846A` | `吃过`、完成态、成员可信标识 |
| 陶土橙 | `#BF715F` | `想吃`、提醒、安排菜单 |
| 柔和边框 | `rgba(91, 74, 59, 0.08)` | 卡片边界 |
| 浅色容器 | `#F1E9DF` | 筛选底、placeholder |

落地建议：

- 新增 `pages/index/styles/tokens.scss` 或在现有组件 SCSS 顶部统一变量。
- P0 先在 `pages/index/components/*.scss` 内用变量替换高频色值。
- 暂不引入远程字体。微信小程序内优先使用系统字体，空间名等重点标题可尝试 `"Songti SC", "STSong", "SimSun", "Noto Serif SC", serif` 系统衬线栈，并用 `font-weight: 700`、行高和颜色塑造接近原型的中文标题气质；统计数字可单独使用 `"Didot", "Baskerville", "Georgia", "Times New Roman", serif`，只影响数字不影响中文身份文案。

### 4.2 圆角与阴影

| 元素 | 建议 |
| --- | --- |
| 页面级卡片 | `24rpx - 28rpx` |
| 列表菜卡 | `24rpx` |
| 封面图 | `18rpx - 20rpx` |
| 胶囊按钮 | `999rpx` |
| 底部浮动导航 | `32rpx - 40rpx` |
| 阴影 | 保持轻量，主卡 `0 12rpx 24rpx rgba(56,44,30,0.05)`，浮动层可更明显 |

不要照搬原型里的过大 `32px/40px` Web 圆角到所有小程序卡片，否则列表密度会下降。

### 4.3 图标

当前使用 `uview-plus` 图标，原型使用 `lucide-react`。落地时保持 `uview-plus`，但需统一语义：

- 美食库：`home/grid/search/reload/calendar`
- 添加：`plus` 或后续 AI 模式下用 `sparkles`
- 空间：`grid/share/edit-pen/user/checkmark-circle`
- 状态：`heart-fill` 表示想吃，`checkmark-circle-fill` 表示吃过

## 5. 美食库页面优化方案

### 5.1 当前问题

| 模块 | 问题 | 影响 |
| --- | --- | --- |
| 顶部信息 | 标题、说明、安排菜单、菜单 spotlight 已拆分，但整体记忆点偏弱 | 用户知道是列表，但不一定理解“美食库 + 菜单安排”的关系 |
| 搜索与筛选 | 搜索、餐别、状态都在一个工具卡里，视觉层级接近 | 首屏控件多，列表内容下移 |
| 状态筛选 | `全部 / 想吃 / 吃过` 没有数量 | 用户无法预判切换后是否有内容 |
| 菜卡 | 已有封面、来源、状态 switch，但列表摘要偏短，状态切换控件语义需要更直观 | 用户可能误触卡片或不理解 switch 两端含义 |
| 空态 | 有说明文案，但缺少明显动作 | 搜索无结果、空分类时容易形成死路 |
| 点菜模式 | 已有浮动结算条，但普通模式与点菜模式的页面层级差异还可以更明显 | “当前正在点菜”状态仍需要更强识别 |
| 添加入口 | 底部中间是加号，原型中是 AI/饮食管家感 | 若后续接 AI，入口语义要提前设计好 |

### 5.2 目标结构

```text
美食库页
├─ 顶部标题区
│  ├─ 图标 + 美食库
│  ├─ 辅助说明 / 同步状态
│  └─ 安排菜单按钮
├─ 菜单 spotlight（仅有记录时展示）
├─ 搜索与筛选面板
│  ├─ 搜索框
│  ├─ 最近搜索 / 推荐词
│  ├─ 餐别 segmented：早餐 / 正餐 + 数量
│  └─ 状态 segmented：全部 / 想吃 / 吃过 + 数量
├─ 列表标题行
│  ├─ 当前筛选摘要
│  ├─ 清除筛选
│  └─ 帮我选
├─ 菜卡列表
└─ 空态 / 无结果态
```

### 5.3 P0 改造项

#### P0-1：搜索与筛选面板视觉收口

落地位置：

- `pages/index/index.vue`
- 建议新增 `pages/index/components/library-toolbar-section.vue`
- 建议新增 `pages/index/components/library-toolbar-section.scss`

改造内容：

- 将当前搜索、搜索建议、餐别筛选、状态筛选从 `index.vue` 模板提取成 `LibraryToolbarSection`。
- 搜索框保持第一行，占据完整宽度，输入后右侧展示清除按钮。
- 餐别和状态仍保持两行，避免小屏横向塞太多。
- 餐别 tab 延续当前 `早餐 / 正餐 + count`。
- 状态 tab 增加数量：
  - `全部`：当前餐别下全部数量
  - `想吃`：当前餐别下 `wishlist` 数量
  - `吃过`：当前餐别下 `done` 数量

验收标准：

- 375px 宽度下 tab 文案不换行、不被挤压。
- 输入搜索词、清除搜索词、点击最近搜索后列表刷新逻辑保持一致。
- 点菜模式下仍隐藏普通筛选，只保留点菜所需搜索。

#### P0-2：空态补动作

落地位置：

- `pages/index/index.vue`
- 可新增 `pages/index/components/library-empty-state.vue`

状态分支：

| 场景 | 标题 | 主动作 | 次动作 |
| --- | --- | --- | --- |
| 搜索无结果 | `没有找到“xxx”` | `清除搜索` | `添加这道菜` |
| 当前餐别为空 | `还没有早餐记录` | `添加早餐` | `切到正餐看看` |
| 当前状态为空 | `正餐里还没有想吃的菜` | `查看全部` | `添加一道正餐` |
| 同步失败且无缓存 | `暂时无法连接空间` | `重试同步` | `去空间页查看` |

当前已有 `emptyStateTitle` / `emptyStateDesc`，建议新增：

- `emptyStatePrimaryActionText`
- `emptyStateSecondaryActionText`
- `handleEmptyStatePrimaryAction`
- `handleEmptyStateSecondaryAction`

验收标准：

- 所有空态至少有一个明确下一步。
- 搜索无结果时不引导用户退出当前页面。
- 同步失败时不误导用户新增到不存在的空间。

#### P0-3：菜卡状态切换可理解性增强

落地位置：

- `pages/index/components/recipe-card-item.vue`
- `pages/index/components/recipe-card-item.scss`

改造内容：

- 保留当前 switch 快捷切换，但扩大可点击区到至少 `64rpx x 48rpx`。
- switch 两端在未选中时也保持轻微可见，避免看起来像单个按钮。
- `card.infoLine` 继续展示 `正餐 · 已整理 / 可整理 / 整理中 / 待重试`。
- 若 `card.listSummary` 为空，但有 `ingredient` 或 `note`，优先展示简短摘要，减少菜卡正文空白。
- 来源 badge 保留在封面左上角；图片数量保留在右下角。

验收标准：

- 点菜模式下点击菜卡主体仍进详情，点击 `加入菜单` 不进详情。
- 普通模式下点击状态 switch 不触发卡片打开。
- 无图菜卡仍保持同等高度，列表不跳动。

#### P0-4：底部导航向浮动胶囊收口

落地位置：

- `pages/index/index.vue` 底部导航样式段

改造内容：

- 保留 DOM 结构和事件：`switchSection('library')`、`openAddSheet`、`switchSection('kitchen')`。
- 将当前全宽底栏改为左右 `24rpx` 内收的浮动胶囊，背景使用 `rgba(255,255,255,0.82)` + `backdrop-filter`。
- 中间加号保持主入口，但增加文字策略：
  - 当前阶段：`添加`
  - 后续 AI 助手上线后：可改为 `灵感` 或 `问问`
- 点菜模式下底部结算条位置保持高于底部导航，避免互相遮挡。

验收标准：

- iPhone 安全区下底部导航不贴边。
- 点菜模式浮动条与底部导航之间至少保留 `12rpx` 间距。
- `美食库` 和 `空间` 激活态明显，未激活态不会显得不可点。

### 5.4 P1 改造项

#### P1-1：顶部标题区强化“今天吃什么”语境

落地位置：

- `pages/index/components/library-header-section.vue`
- `pages/index/components/library-header-section.scss`

改造内容：

- 标题仍为 `美食库`，说明文案保留 `按餐别整理，想吃和吃过更清楚`。
- 安排菜单按钮保持右侧，但视觉上使用陶土橙轻强调，而不是和普通筛选同权重。
- 如果存在近期菜单记录，spotlight 改为更接近原型 `PlanCard`：
  - 左侧日期与状态
  - 中间菜品摘要
  - 右侧 `1/3` 或箭头
- 如果没有菜单记录，继续隐藏 spotlight，符合 2026-04-25 的既有决策。

验收标准：

- 无菜单记录时不展示空卡。
- 有多个菜单记录时滑动切换和点击进入详情保持不变。

#### P1-2：列表标题行提升效率

落地位置：

- `pages/index/index.vue`

改造内容：

- 当前 `正餐 · 想吃 · 搜“xxx” · 3道` 保留。
- 当列表数量大于 `0` 时展示 `帮我选`；空态时把随机推荐入口移入空态逻辑，不占标题行。
- `清除` 只在搜索词或状态筛选非默认时展示。

验收标准：

- 搜索中、筛选中、默认态三种情况下标题行信息不换行到两行。

#### P1-3：添加入口与 AI 辅助预留

当前不建议直接把原型的 AI Chat Sheet 照搬进小程序。可先保留 `add-recipe-sheet`，但在文案和入口上为“贴链接/智能识别”服务。

改造建议：

- 底部中间按钮仍打开 `add-recipe-sheet`。
- `add-recipe-sheet` 顶部突出“贴小红书/B站链接自动识别菜名”的能力。
- 搜索无结果时的 `添加这道菜` 可打开新增弹层，并把搜索词预填到 `draft.title`。
- 未来如果增加 AI 助手，再新增独立 `ai-assistant-sheet`，不要复用新增菜谱弹层承载聊天。

验收标准：

- 不新增后端接口也能完成 P1。
- 用户从无结果态添加时，菜名能自动带入。

## 6. 空间页面优化方案

### 6.1 当前问题

| 模块 | 问题 | 影响 |
| --- | --- | --- |
| 当前空间卡 | 已有空间名、身份、成员数、空间数，但整体更像信息卡 | “切换、编辑、连接状态”三个行为混在同一卡片里 |
| 邀请入口 | 已有主按钮，但和当前空间卡视觉分离较大 | 用户可能先看成员列表，忽略邀请 |
| 成员列表 | 当前只展示成员身份和说明 | 缺少“最近协作/共同维护”的空间感 |
| 加入邀请码 | 入口在底部，存在但弱 | 被邀请用户或多空间用户不容易发现 |
| 空间统计 | 当前只有成员/身份/空间数 | 对“这个空间里有多少菜、多少想吃/吃过、最近安排”缺少反馈 |

### 6.2 目标结构

```text
空间页
├─ 当前空间概览卡
│  ├─ 当前空间 badge + 连接状态
│  ├─ 空间名称 + 编辑
│  ├─ 一句话说明
│  ├─ 统计格：成员 / 身份 / 空间数
│  └─ 切换空间入口
├─ 协作动作区
│  ├─ 邀请成员（主按钮）
│  └─ 输入邀请码加入（次入口）
├─ 空间内容概览（P1）
│  ├─ 美食库总数
│  ├─ 想吃 / 吃过
│  └─ 最近菜单
├─ 成员列表
└─ 关于我们
```

### 6.3 P0 改造项

#### P0-1：当前空间卡层级重排

落地位置：

- `pages/index/components/kitchen-section.vue`
- `pages/index/components/kitchen-section.scss`

改造内容：

- 卡片顶部左侧：`当前空间` badge。
- 卡片顶部右侧：
  - 有多个空间时显示 `切换 >`
  - 只有一个空间时显示连接状态 `已连接` 或 `未连接` + 状态点
- 空间名称单独一行，编辑按钮贴近空间名。
- 统计格保持三列，但视觉上减少边框，改成半透明白底。
- 当 `currentKitchenName` 为空时，展示明确失败/同步态，并弱化统计区。

验收标准：

- 空间名过长时最多两行，编辑按钮不被挤出。
- `切换` 与 `编辑` 是两个独立点击目标，不互相误触。
- 没有空间时不展示“成员/身份/空间”的无意义数字。

#### P0-2：邀请与加入入口组合

落地位置：

- `pages/index/components/kitchen-section.vue`

改造内容：

- 邀请成员主按钮保留深色主按钮样式。
- `已有邀请码？去加入` 从成员列表底部上移到邀请区域下方，作为次级轻按钮。

## 7. 组件拆分建议

当前 `pages/index/index.vue` 已经承担大量状态与业务逻辑，继续把 UI 片段拆出去会更利于后续重构。

| 新/现有组件 | 类型 | 责任 |
| --- | --- | --- |
| `library-header-section` | 现有，调整 | 美食库标题、安排菜单、菜单 spotlight |
| `library-toolbar-section` | 新增 | 搜索、最近搜索、餐别、状态筛选 |
| `library-empty-state` | 新增 | 美食库空态、搜索无结果、同步失败动作 |
| `recipe-card-item` | 现有，调整 | 菜卡视觉、状态切换、点菜模式加入菜单 |
| `kitchen-section` | 现有，调整 | 空间页整体 |
| `space-overview-panel` | 可选新增 | 如果 P1 空间内容概览较复杂，再从 `kitchen-section` 拆出 |

拆分原则：

- 业务状态仍在 `index.vue`，组件只接收 props 和 emit 事件。
- 不在子组件内直接调用 API，避免状态来源分散。
- 新增组件的样式文件与组件同目录，延续当前风格。

## 10. 风险与处理

| 风险 | 说明 | 处理方式 |
| --- | --- | --- |
| 原型 Web 化过强 | 原型使用 React、Tailwind、Lucide、Google Fonts 和远程图片 | 小程序落地只取结构和视觉语言，不照搬依赖 |
| 首屏控件过多 | 美食库天然包含搜索、筛选、菜单、列表 | P0 通过面板收口和列表标题轻量化处理 |
| 空间页变重 | 加入空间内容概览后可能过长 | P1 概览保持 2x2 小卡，不做大型统计面板 |
| 点菜模式被影响 | 美食库重构可能破坏点菜模式条件渲染 | 所有筛选改造必须保留 `isLibraryMealOrderMode` 分支 |
| 色彩一体化不足 | 当前棕色系较多，原型也偏棕 | 建立 token，并强制状态色参与关键反馈 |

## 11. 推荐优先级

优先顺序建议：

1. `P0-1` 搜索筛选组件化与状态数量。
2. `P0-2` 空态动作补齐。
3. `P0-3` 菜卡状态切换与摘要优化。
4. 空间页 P0：当前空间卡、邀请/加入入口、成员卡。
5. 美食库 P0：底部导航浮动胶囊。
6. `P1` 空间内容概览和无结果预填新增。

这套顺序能先解决可用性和信息层级，再处理更强的视觉表达，风险比一次性重绘整页更低。
