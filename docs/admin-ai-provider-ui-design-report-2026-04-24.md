# Admin Web AI Provider UI 界面设计报告

> 评估对象：`admin-web/src/pages/AIProvidersPage.vue`
> 评估页面：`/admin/ai-providers`
> 评估时间：2026-04-24
> 评估依据：当前页面截图、现有实现、`ui-ux-pro-max` 的 Data-Dense Dashboard / Drill-Down 后台页建议

本文聚焦 `admin-web` 的 `AI Provider` 模块。该模块不是普通配置表单，而是同时承担
**多 Provider 路由配置、草稿测试、密钥维护、故障恢复、审计追溯** 的后台运维控制台。
因此界面设计目标应从“展示配置项”升级为：

- 让管理员 3 秒内判断当前线上链路是否安全。
- 让配置、测试、保存、回滚的主流程清晰可控。
- 在 Provider 数量增长时保持高效扫描和低误操作率。
- 把审计与测试结果变成可用于排障的证据，而不是长表格堆叠。
- 整体界面保持简洁明了：默认只显示操作必需信息，解释说明通过 hover tips、
  click popover 或详情抽屉渐进披露。

---

## 1. 设计基准

### 1.1 推荐设计模式

`ui-ux-pro-max` 对该页面匹配到的推荐模式是：

- **模式**：Data-Dense + Drill-Down
- **风格**：Data-Dense Dashboard
- **交互策略**：Progressive disclosure，默认只显示每一步必要信息，补充说明按需展开
- **适用场景**：后台配置、运维面板、数据密集型控制台
- **关键交互**：
  - hover tooltip
  - row highlighting
  - smooth filter feedback
  - loading / success / error 状态明确
  - 表格可横向滚动或转为卡片布局

这与当前页面高度吻合：页面不需要营销式视觉，也不适合大面积插画或低密度卡片。
它需要更像一个稳定、克制、可扫描的生产控制台。

### 1.2 简洁优先与 Tips 策略

本页面后续改造应采用“默认简洁、按需解释”的信息展示原则。

| 信息类型 | 展示方式 | 示例 |
| --- | --- | --- |
| 必须立即判断的状态 | 常驻短文案 / tag | `已同步`、`草稿 3 项`、`测试异常` |
| 普通字段解释 | hover tooltip | `maxAttempts` 推荐值、`invalid_response` 含义 |
| 多条件规则 | click popover | 草稿/正式/兼容链路状态矩阵 |
| 复杂结果 | drawer / expandable row | 审计 diff、测试 attempts 详情 |
| 高风险动作 | modal confirmation | 保存 diff、清空密钥、删除 Provider |

默认页面上不应出现长段解释文字。除非是错误、风险、空态或保存影响，否则说明应进入
tooltip / popover。这样能保持页面干净，同时不牺牲新用户理解成本。

### 1.3 当前页面角色

当前页面承载三类任务：

| 任务类型 | 典型动作 | 风险等级 |
| --- | --- | --- |
| 配置维护 | 开关新路由、调整策略、修改超时、增删 Provider | 高 |
| 故障排查 | 单节点测试、草稿测试、查看测试结果 | 中高 |
| 追溯审计 | 查看旧值/新值、筛选保存或测试记录 | 中 |

所以设计上应优先保证：

- 主流程可见：编辑 -> 测试 -> 保存。
- 风险操作可确认：删除节点、清空密钥、保存影响线上。
- 结果可追踪：测试结果、审计 diff、操作人、时间。
- 状态不靠颜色单独表达：必须同时使用文字、icon、tag。

## 2. 当前页面优点

### 2.1 主功能完整

当前实现已经覆盖核心运维功能：

- 支持 `summary / title / flowchart` 三个场景切换。
- 支持草稿编辑、草稿测试、保存、放弃草稿。
- 支持 Provider 启停、排序、复制、删除、单节点测试。
- 支持 API Key 的保留、更换、清空语义。
- 支持未保存草稿提示与离开保护。
- 支持路由审计筛选和分页。

这说明页面的信息基础已经比较扎实，后续优化不需要大改业务模型。

### 2.2 风险意识已经建立

截图与代码中已有多处风险控制：

- `保存场景` 在无改动时禁用。
- `放弃草稿` 有 tooltip 和确认。
- `清空密钥` 先标记，保存后才真正生效。
- `单节点测试` 与 `草稿测试` 分离。
- `线上链路` 用 `summary-v2 / title-compat / flowchart-draft` 显示生效路径。

这些都是后台配置页很关键的“防事故”基础。

### 2.3 可访问性基础较好

当前页面已经有一些正确方向：

- 场景卡使用 `role="tablist"` / `role="tab"`。
- 图标按钮有 `aria-label`。
- 折叠按钮有 `aria-expanded`。
- loading、empty、error 状态由 `PageState` 统一承载。
- 表格外层有 `table-scroll`，具备横向滚动基础。

后续优化应继续沿着这个方向补齐，而不是只做视觉美化。

## 3. 主要问题

### 3.1 首屏信息层级偏散

页面首屏同时出现：

- 三张场景卡。
- 告警配置提示条。
- 当前场景面包屑。
- 左侧场景策略。
- 右侧 Provider 节点。
- 顶部 toolbar 操作。

这些信息都重要，但权重接近，导致用户需要自己判断“现在最该看哪里”。

当前最该优先表达的其实只有三件事：

1. 当前线上链路是什么。
2. 当前草稿是否有改动。
3. 当前 Provider 配置是否可测试/可保存。

建议把首屏的核心信号收敛为一个明确的“当前场景状态条”，其它提示下沉。

### 3.2 场景卡更像导航，不像健康概览

三张场景卡展示了策略、节点数、来源、最近修改和线上链路，但缺少真正的运行健康信号：

- 最近一次测试是否成功。
- 最近一次失败类型。
- 是否有节点在冷却或熔断。
- 近 1 小时成功率 / 平均耗时。
- 最后一次保存是否由当前账号操作。

对于运维页面，场景卡最好承担“全局概览”职责，而不只是大号 tab。

### 3.3 Provider 卡片在节点增多后会变重

当前每个 Provider 是完整表单卡片。1-2 个节点时清晰，但当节点达到 4 个以上时会出现：

- 页面纵向滚动很长。
- 每个节点重复展示大量字段。
- 用户想比较节点差异时，需要上下滚动。
- 排序、启停、测试这些高频动作被埋在大卡片顶部。

更适合的模式是“紧凑列表 + 展开详情”：

- 默认行展示名称、id、model、endpoint、timeout、启用状态、测试状态。
- 点击行或折叠按钮展开完整表单。
- 多节点时仍能一屏扫描主要状态。

### 3.4 测试结果离触发点偏远

页面已有 `最近测试结果` 卡片，但它在编辑区下方。用户点击顶部 `测试当前草稿`
或 Provider 行内测试后，视觉焦点仍留在表单区。

这会造成两个问题：

- 测试完成后不知道结果在哪里。
- 单节点测试结果和草稿测试结果容易被后续测试覆盖，缺少上下文。

建议把测试结果拆成两层：

- 顶部轻量结果 chip：成功/失败、最终节点、耗时、可点击跳转详情。
- 下方详情卡：attempts 表格、错误类型、备注、可复制 requestId 或 trace 信息。

### 3.5 审计区太重，且可读性不足

审计表直接铺在页面下方，字段很多，但核心 diff 信息被 `show-overflow-tooltip` 截断。

当前审计区的问题：

- 默认占用大量纵向空间。
- `oldValueMasked / newValueMasked` 不易比较。
- 只能按动作筛选，缺少时间、操作人、配置键搜索。
- 行不可展开查看结构化 diff。

更合适的交互是：

- 默认只展示最近 5 条，完整审计进入抽屉或独立区。
- 审计行点击展开，左右对比旧值/新值。
- 对保存记录提供“复制为草稿”或“复制配置键”能力。

### 3.6 常驻告警提示打断主流程

“连续异常邮件告警已接入配置中心”是有价值的信息，但它不是每次进入页面都必须处理的阻塞项。
当前横幅常驻在首屏，容易稀释真正的风险提示。

建议：

- 如果告警未配置或配置异常，再显示 warning/info alert。
- 如果只是入口提示，放到右上角“告警配置”按钮或当前场景状态条的次级入口。
- 如果近期真的触发过告警，则显示最近告警时间和 Provider。

### 3.7 视觉密度与圆角阴影偏展示型

页面整体卡片圆角、阴影、渐变较多，观感温和，但作为后台运维控制台略显“展示型”。

建议调整方向：

- 卡片半径控制在 8-12px。
- 减少大面积阴影和渐变，更多依赖分隔线和状态色。
- 对表单、列表、审计使用更稳定的行高和网格密度。
- 主 CTA 与次 CTA 保持清晰层级。

### 3.8 长页面缺少持久状态与动作反馈

参考 CPAMC 的底部浮动 bar，可以看到一个有效模式：

- 配置已加载时显示 `配置已加载` + 刷新 + 完成 icon。
- 有未保存更改时显示 `有未保存的更改` + 刷新 + 带红点的保存/完成 icon。

这个模式的价值不在视觉皮肤，而在交互语义：

- 用户滚动到页面中下部时，仍能知道当前是否有未保存更改。
- 高频动作不必回到顶部 toolbar 才能触发。
- 状态与动作绑定在一起，比单独 alert 更像“当前编辑会话状态”。

当前 `AI Provider` 页面虽然顶部已有保存、测试、放弃草稿，但页面很长，尤其 Provider
和审计区展开后，用户在页面中下部容易失去“现在能不能保存、是否已同步”的感知。
底部浮动 bar 可以作为顶部 toolbar 的轻量镜像，增强持续反馈。

### 3.9 说明文案需要进一步收敛

当前页面已经把很多风险解释直接写在面板副标题、helper text、alert 和备注区中。
这些文案对新用户有帮助，但常驻展示会降低老用户的扫描效率，也会让页面显得更重。

建议把说明分级处理：

- **常驻保留**：状态、字段 label、当前值、错误/警告、保存影响。
- **移入 tooltip**：推荐区间、字段含义、错误类型解释、icon 操作含义。
- **移入 popover**：线上链路状态矩阵、调度策略差异、图片接口参数规则。
- **移入抽屉/展开区**：审计 diff、完整测试 attempts。

尤其是 `场景策略`、`Provider 节点`、`场景级请求参数` 这类面板副标题，可以压缩成
短标题 + `Info` icon。鼠标 hover 后再显示完整说明。

### 3.10 缺少统一的 Tips、校验与微状态体系

继续按 `ui-ux-pro-max` 的极简后台配置页口径看，当前还缺少三类“系统化小能力”：

- **统一 Tips 组件**：如果每个字段临时写 `el-tooltip`，后续会出现 icon 尺寸、文案长度、触发方式和无障碍行为不一致。需要一个统一的 `HelpTip` 组件承载短解释，并约束复杂说明升级为 popover。
- **字段级前置校验**：目前更多依赖保存/测试时反馈，但配置页应在 blur 时尽早发现问题，例如 `Provider ID` 唯一性、`Base URL` 格式、`Model` 非空、超时范围异常。
- **行内微状态**：Provider 行需要显示最近测试状态、耗时、失败类型等短信息，否则用户仍要滚到测试详情才能判断节点是否可靠。

这些能力不会让界面变复杂，反而能减少常驻说明、减少错误 toast、减少用户来回滚动。

### 3.11 Tips 不能隐藏阻塞与高风险状态

“说明进 tips”不能理解为“所有提示都隐藏”。以下信息必须常驻可见：

- 没有启用 Provider。
- 所有 Provider 都在冷却或熔断。
- Provider 缺 API Key。
- `maxAttempts` 大于启用节点数。
- 测试失败或最近一次测试异常。
- 保存会清空密钥、删除节点或影响线上链路。

这些属于阻塞/高风险状态，必须用短文案、tag、inline error 或确认弹窗直接呈现。
tooltip 只能解释“为什么”和“如何处理”，不能作为唯一告警载体。

## 4. 优化方案

### 4.1 信息架构重排

建议页面结构调整为：

```text
顶部 toolbar
  刷新 / 放弃草稿 / 测试草稿 / 保存场景 / 账号

场景状态区
  compact tabs: 正文总结 / 标题清洗 / 步骤图生成
  当前线上链路、草稿状态、最近测试结果、告警入口；复杂解释进入 tooltip/popover

主编辑区
  左侧：场景策略 sticky panel
  右侧：Provider 紧凑列表 + 展开编辑

底部浮动会话 bar
  已同步/有草稿/测试中/测试异常 + 刷新 + 测试 + 保存

辅助区
  最近测试详情
  最近审计 5 条
  完整审计抽屉或独立面板
```

这样可以把“配置主线”和“排障辅助信息”分开，降低首屏负担。

### 4.2 场景卡改为紧凑状态 tab

当前三张场景卡可以保留，但建议降低尺寸，强化状态字段，并把解释性信息收进 tips。

推荐字段：

| 字段 | 展示方式 | 目的 |
| --- | --- | --- |
| 场景名称 | 主标题 | 快速定位 |
| 生效链路 | `summary-v2` / `summary-compat` tag | 明确线上路径 |
| 节点状态 | `1/1 启用` | 判断是否有可用节点 |
| 最近测试 | success/warning tag | 发现异常 |
| 最近修改 | tooltip 或小字 | 追踪配置新鲜度 |

可选增强：

- 近 1 小时成功率。
- 平均耗时。
- 最近一次失败类型。
- 熔断中节点数。

应收进 tooltip / popover 的信息：

- `来源：数据库覆盖`
- `策略：轮询起始 + 失败切换`
- `最近修改人 / 完整时间`
- `summary-v2 / summary-compat / summary-draft` 的含义

### 4.3 当前场景状态条

在场景 tab 下方增加一条状态条：

```text
正文总结 · summary-v2 · 已同步 · 最近测试通过
```

状态条职责：

- 替代当前面包屑中过多分散的信息。
- 解释草稿与线上链路的关系。
- 作为测试结果和告警配置入口。

当 `isDirty=true` 时状态条变为：

```text
正文总结 · summary-draft · 草稿 3 项 · 保存后影响线上
```

这比单独一个黄色 alert 更贴近用户决策。

状态条不要塞长解释。`测试仅作用于草稿`、`summary-draft` 含义、`保存后影响线上`
的具体规则都应通过 `Info` tooltip / popover 展示。

### 4.4 底部浮动状态操作条

底部浮动 bar 可以借鉴 CPAMC，但要按 `AI Provider` 的高风险配置场景重新定义。

推荐定位：

- 它是“当前编辑会话状态 + 高频动作”的持久入口。
- 它不替代顶部 toolbar，也不绕过保存前 diff、清空密钥确认等安全机制。
- 桌面端默认居中悬浮在底部；移动或窄屏时可贴底并留出 safe area。

推荐状态：

| 状态 | 文案 | 视觉 | 说明 |
| --- | --- | --- | --- |
| 已同步 | `已同步` | neutral/success chip | 当前草稿与远端一致 |
| 有草稿 | `有未保存的更改 · 3 项` | warning chip + 保存红点 | 对应 `isDirty` 与 `diffCount` |
| 测试中 | `测试中...` | loading chip | 草稿或单节点测试正在进行 |
| 测试通过 | `测试通过 · summary-compat` | success chip | 点击跳到测试结果 |
| 测试异常 | `测试异常 · timeout` | warning chip | 点击跳到测试结果 |

推荐动作：

| 动作 | icon | 规则 |
| --- | --- | --- |
| 刷新远端 | Refresh | 无草稿时直接刷新；有草稿时弹出“保存 / 放弃 / 取消” |
| 测试草稿 | Promotion / Play | loading 时禁用；结果写入 bar 与测试详情 |
| 保存场景 | Check | 仅 `isDirty=true` 可用；点击后先展示 diff 确认 |

状态示例：

```text
已同步        [刷新] [测试] [完成]
有未保存的更改 · 3 项  [刷新] [测试] [保存 · 红点]
测试中...     [刷新 disabled] [测试 loading] [保存 disabled]
测试异常 · timeout  [刷新] [重测] [保存]
```

实施注意：

- 不要只用红点表达未保存，必须同时有 `有未保存的更改` 文案。
- icon-only 按钮必须有 tooltip 和 `aria-label`。
- bar 不能遮挡审计分页、表格横向滚动条和页面底部内容，页面底部需要增加 padding。
- `刷新` 在有草稿时不能直接丢弃本地改动，必须复用现有 `guardUnsavedChanges`。
- `保存` 不能因为入口变轻而降低风险控制，仍然要走保存前 diff。

### 4.5 Provider 节点改为列表 + 展开编辑

推荐默认行：

| 顺序 | 节点 | Model | Endpoint | Timeout | 密钥 | 状态 | 操作 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 1 | 兼容单节点 `summary-compat` | gemini-flash-latest | chat/completions | 30s | 已配置 | 开 | 测试 / 展开 / 更多 |

展开后再显示完整表单：

- Provider ID
- 展示名称
- Base URL
- Model
- Endpoint
- 超时
- Response Format
- API Key 管理

好处：

- 节点越多，收益越大。
- 用户可以横向比较 Provider 差异。
- 高频操作不再被大表单稀释。
- 更接近后台运维工具的扫描习惯。

Provider 区的说明收敛建议：

- 行内只显示关键字段：名称、id、model、endpoint、timeout、密钥状态、启用状态。
- `adapter`、`responseFormat`、`priority` 这类低频字段默认不常驻，展开后显示。
- 非图片节点直接隐藏 `Response Format`，不要展示 disabled 控件。
- `images/generations` 的 `output_format=png / response_format` 解释放到字段旁 `Info` tooltip。
- API Key 只常驻显示状态 chip：`已保存密钥`、`待清空`、`未配置`；“留空保留旧值”的解释放在 tooltip 或点击“更换密钥”后的局部提示中。

### 4.6 测试交互优化

建议把测试流程设计为：

1. 点击 `测试当前草稿`。
2. 按钮进入 loading。
3. 顶部状态条显示 `测试中...`。
4. 完成后显示：
   - 成功：`测试通过 · via summary-compat · 30s`
   - 失败：`测试异常 · timeout · 查看详情`
5. 自动滚动到测试详情，或提供明显的 `查看详情`。

单节点测试建议在 Provider 行内显示局部结果：

- `通过 1.2s`
- `失败 timeout`
- `冷却中`

全局测试结果卡保留 attempts 明细，但不应是唯一反馈入口。

测试区的简洁原则：

- 顶部/底部 bar 只显示一句结果：`测试通过 · summary-compat · 30s`。
- 失败时常驻显示错误类型，完整错误信息放入 tooltip 或详情卡。
- attempts 表格默认展示核心列：Provider、状态、耗时、错误类型；备注长文本用 tooltip。
- `timeout / rate_limit / invalid_response` 等术语使用 label + tooltip，不在表格里铺长说明。

### 4.7 保存前差异确认

AI Provider 配置会影响线上链路，保存前建议增加 diff dialog。

保存前弹窗内容：

```text
即将保存「正文总结」场景

场景参数
- maxAttempts: 2 -> 1
- cooldownSeconds: 60 -> 30

Provider
- summary-compat.timeoutSeconds: 30 -> 60
- 新增 provider: summary-backup
- 清空密钥: summary-old

保存后影响线上链路：summary-v2
```

按钮：

- 取消
- 返回编辑
- 确认保存

这能显著降低误保存和误清空密钥风险。

### 4.8 审计区改造

建议分两层：

#### 最近审计

页面底部默认展示最近 5 条：

- 时间
- 操作人
- 动作
- 配置键
- 摘要

#### 完整审计

点击 `查看完整审计` 打开抽屉或独立面板：

- 动作筛选
- 操作人筛选
- 时间范围
- 配置键搜索
- page size 切换
- 行展开 diff

审计 diff 需要从“字符串 tooltip”升级为“结构化对比”。

审计区默认也应保持简洁：

- 默认只展示最近 5 条，不直接铺完整表格。
- 旧值/新值列默认显示摘要，例如 `maxAttempts 2 -> 1`。
- 完整 masked 值和 JSON diff 放到展开行或抽屉。
- settingKey 较长时只显示尾部关键段，完整值 hover tooltip 展示。

### 4.9 Tips 与渐进披露规范

为了统一后续实现，建议建立一套 tooltip / popover 规范。

#### 字段级 tooltip

适用于短解释，hover 即可读完：

| 字段 | 常驻展示 | Tooltip 内容 |
| --- | --- | --- |
| 最大尝试次数 | `最大尝试次数` | `建议 2-5 次；过大会拖慢失败降级` |
| 熔断阈值 | `熔断阈值` | `连续失败达到该次数后暂时跳过该节点` |
| 冷却时间 | `冷却时间` | `低于 30s 容易抖动，超过 5 分钟恢复较慢` |
| Response Format | `Response Format` | `仅 images/generations 生效；output_format 固定为 png` |

#### 术语级 tooltip

适用于错误类型、策略和链路状态：

| 术语 | Tooltip 内容 |
| --- | --- |
| `timeout` | 上游请求超过当前 Provider 超时时间 |
| `rate_limit` | 上游返回限流或配额相关错误 |
| `invalid_response` | 上游响应格式不符合当前解析预期 |
| `summary-draft` | 当前仅草稿测试生效，线上仍使用上次保存配置 |
| `compat` | 兼容旧单 Provider 链路 |

#### Popover

适用于一屏内能说明清楚的规则：

- `线上链路` tag：点击展示草稿/正式/兼容链路矩阵。
- `调度策略`：hover 只显示一句话，点击可展示策略差异。
- `告警配置`：点击展示当前阈值、SMTP 是否配置、收件人数量。

#### Drawer / 展开区

适用于复杂信息：

- 测试 attempts 完整详情。
- 审计 old/new JSON diff。
- Provider 高级参数。

#### Modal

只用于阻塞式高风险确认：

- 保存场景前 diff。
- 清空密钥。
- 删除 Provider。
- 有草稿时刷新或切换场景。

### 4.10 HelpTip 与说明配置中心

建议新增一个统一的 `HelpTip` 组件，作为字段解释、术语解释、icon-only 操作说明的唯一入口。

设计要求：

- 默认使用统一的 `Info` icon，尺寸与表单 label 对齐。
- hover 与 keyboard focus 都能触发。
- tooltip 最大宽度固定，文案控制在 1-2 行。
- 复杂内容不放 tooltip，改用 popover 或 drawer。
- 支持 `aria-label`，并避免纯鼠标用户专属交互。

说明文案建议集中维护，不散落在模板中：

| key | 用途 |
| --- | --- |
| `field.maxAttempts` | 最大尝试次数推荐值与风险 |
| `field.failureThreshold` | 熔断阈值解释 |
| `field.cooldownSeconds` | 冷却时间解释 |
| `field.responseFormat` | 图片生成返回格式解释 |
| `term.timeout` | timeout 错误类型 |
| `term.rateLimit` | rate_limit 错误类型 |
| `term.invalidResponse` | invalid_response 错误类型 |
| `channel.draft` | draft 链路说明 |
| `channel.compat` | compat 链路说明 |

集中维护的好处是术语一致、可复用，也便于以后国际化或批量调整文案。

### 4.11 字段校验与阻塞状态

配置页应采用“blur 校验 + 保存前兜底”的策略。常规错误在字段旁边显示，保存时再做完整校验。

建议校验规则：

| 字段 | 校验 | 展示 |
| --- | --- | --- |
| Provider ID | 非空、同场景唯一、稳定字符集 | 字段 inline error |
| Base URL | 非空、合法 URL、建议以 `/v1` 结尾但不强制 | inline error + HelpTip |
| Model | 非空 | inline error |
| Timeout | 1-600 秒；长耗时场景提示推荐区间 | inline warning |
| Endpoint | `images/generations` 时才展示 Response Format | 动态字段 |
| API Key | 新节点必须录入或明确标记稍后配置 | 节点行 warning |

阻塞状态建议直接出现在 Provider 行或场景状态条：

- `未配置密钥`
- `无启用节点`
- `所有节点冷却中`
- `保存将清空 1 个密钥`

### 4.12 底部 Bar 草稿摘要 Popover

底部浮动 bar 的 `草稿 N 项` 不应只是状态文本，建议可点击打开摘要 popover。

示例：

```text
草稿 4 项

场景参数 2 项
- maxAttempts: 2 -> 1
- cooldownSeconds: 60 -> 30

Provider 1 项
- summary-compat.timeoutSeconds: 30 -> 60

密钥变更 1 项
- summary-old: 待清空

[查看保存 diff]
```

这样既保持 bar 简洁，又能让用户在任何滚动位置快速知道“到底改了什么”。

### 4.13 Provider 最近测试微状态

Provider 紧凑行应增加最近测试微状态，避免用户必须下滚到测试详情。

推荐状态：

| 状态 | 展示 |
| --- | --- |
| 未测 | `未测` neutral |
| 测试中 | `测试中...` loading |
| 通过 | `通过 1.2s` success |
| 失败 | `失败 timeout` warning |
| 冷却中 | `冷却中` warning |

状态点击后可跳到最近测试详情，失败状态 hover 展示完整错误信息。

### 4.14 长耗时测试计时反馈

`flowchart` 的图片生成链路可能持续几十秒到 3 分钟。单纯 loading 容易让用户误判卡死。

建议测试中显示：

```text
测试中 · 已等待 42s · 预计最长 180s
```

超过 60 秒时，可通过 tooltip 解释：

```text
图片生成链路通常耗时更长；线上代理与后端测试接口当前按 300s 口径等待。
```

注意不要把长解释常驻铺开，常驻只保留计时与状态。

### 4.15 锚点目录按需出现

参考 CPAMC 的左侧锚点目录是有价值的，但不建议无条件常驻，否则会让简洁页面变重。

建议满足任一条件时展示：

- Provider 数量 >= 3。
- 测试结果已存在。
- 审计区展开。
- 页面内容高度超过一屏半。

否则保持当前主内容结构，不额外增加左侧目录。

## 6. 视觉规范建议

### 6.1 密度

- 页面级容器：保持当前宽度即可。
- 卡片 padding：从 22px 收敛到 16-20px。
- 卡片圆角：建议 8-12px，避免大面积 18px。
- Provider 默认行高：48-56px。
- 展开表单字段间距：12-14px。
- 底部浮动 bar：高度 48-56px，胶囊形但不过度圆润；内容区底部增加 72-88px padding，避免遮挡。
- 面板副标题默认不超过一行；超过一行的说明移入 `Info` tooltip。
- helper text 仅在 warning/error 或用户正在编辑该字段时显示。

### 6.2 色彩

当前蓝色主色适合后台，不建议大换色。重点是减少同色系泛用。

建议：

- 主操作：蓝色。
- 成功/正式：绿色。
- 草稿/测试：蓝色或青色。
- 兼容/风险：橙色。
- 危险操作：红色，仅在确认或菜单中出现。

不要只用颜色表达状态，必须配合文字，例如：

- `正式模式`
- `兼容链路`
- `草稿未保存`
- `测试异常`

### 6.3 字体与数字

- 配置键、Provider ID、模型名继续使用 monospace。
- 普通说明文字不要过浅，保证至少 4.5:1 对比度。
- 时间、耗时、节点顺序建议右对齐或使用等宽数字，便于扫描。

### 6.4 动效

- hover 使用颜色、边框、轻阴影即可。
- 避免过多 `translateY`，后台表单中轻微位移会让界面显得不稳定。
- loading、保存成功、测试失败需要明确状态反馈。
- 底部浮动 bar 的出现/状态切换用 150-220ms opacity/transform 即可，避免弹跳动画。
- Provider 行测试状态与底部 bar 计时更新不应造成布局宽度跳动，建议使用固定宽度数字区或等宽数字。

### 6.5 Icon 与符号

- 不使用 emoji 作为状态符号，例如不使用 `📝 3 项未保存`。
- 状态 icon 统一来自 Element Plus Icons 或当前后台已采用的图标库。
- icon-only 操作必须同时具备 tooltip、`aria-label`、focus ring。
- 同类 icon 保持一致尺寸、线宽和按钮面积，避免刷新、测试、保存在不同区域视觉重量不一致。

## 7. 可访问性检查点

后续改造时必须保留或补齐：

- 所有图标按钮必须有 `aria-label`。
- 表单字段必须有可感知 label，不能只靠 placeholder。
- 错误信息使用 `role="alert"` 或可被读屏感知的组件。
- 状态不能只靠颜色表达。
- 表格在窄屏下必须支持横向滚动，或切换为卡片布局。
- 场景 tab 支持键盘左右切换。
- 保存、测试、删除、清空密钥等关键操作可以通过键盘完成。
- 底部浮动 bar 内的 icon-only 操作必须有 tooltip、`aria-label` 和键盘 focus ring。
- tooltip 不能承载必须立即知道的错误；错误和阻塞状态仍要常驻显示并可被读屏感知。
- hover tooltip 需同时支持键盘 focus 触发，避免只对鼠标用户可见。
- 提供 skip link 或等价机制，让键盘用户可跳过侧边导航直接进入主编辑区。
- 自定义 tab / popover / bottom bar 的 Tab 顺序必须与视觉顺序一致，避免键盘陷阱。

| 验收项 | 标准 |

| 首屏判断 | 管理员 3 秒内能说出当前场景线上链路、是否有草稿、最近测试是否正常 |
| 简洁度 | 首屏不出现长段说明；解释性内容主要通过 tips / popover 按需展示 |
| 测试反馈 | 点击测试后无需手动找结果，完成状态在顶部可见 |
| 长耗时反馈 | flowchart 测试中可看到已等待时长和预计最长等待口径 |
| 持久反馈 | 滚动到 Provider 或审计区底部时，仍能看到已同步/有草稿/测试异常状态 |
| 字段校验 | Provider ID、Base URL、Model、Timeout 的错误能在保存前暴露 |
| 多节点效率 | 5 个 Provider 时仍能一屏扫描节点名称、状态、模型、超时 |
| 保存安全 | 保存前能看到关键 diff，清空密钥必须二次确认 |
| 审计可用 | 任意保存记录能展开看到旧值/新值差异 |
| 响应式 | 1366px、1440px、移动窄屏不出现布局错乱或文本重叠 |
| 无障碍 | icon-only 操作均有 tooltip 与 aria-label |

## 9. 推荐后续实施顺序

基于 `2026-04-25` 的复核结果，建议后续优先处理仍未完成且对日常使用最有帮助的项：

1. 完整审计补齐筛选：动作之外，再加操作人、时间范围、配置键搜索和 page size。
2. 增加 `skip link` 或等价“跳到主编辑区”入口，补齐当前无障碍短板。
3. 为高风险阻塞状态做一次专项回归，确认不会只通过 tooltip 表达。
4. 长页面按条件展示锚点目录，先覆盖 Provider 数量多、测试结果存在、审计抽屉打开等场景。
5. 场景卡补最近测试状态；若后端数据充足，再逐步补健康指标。
6. 把最近告警状态和场景状态条联动，避免只有“前往配置”入口而缺风险信号。

再往后再考虑长期增强：

1. 快捷键：保存、测试、切场景。
2. 暗色模式。
3. 沙盒回放最近真实请求。
4. 更完整的告警事件时间线。

## 10. 总结

当前 `AI Provider` 页面已经具备完整后台配置能力，安全意识也比普通配置页强。
下一步最值得投入的不是换一套视觉风格，而是把它从“卡片式配置页面”升级成
“生产路由控制台”：

- 首屏突出线上链路、草稿、测试状态。
- 默认界面保持简洁，说明文案通过 tooltip / popover / drawer 渐进披露。
- 通过统一 HelpTip、字段 blur 校验和行内微状态，把说明和错误前置但不铺满页面。
- Provider 从大卡片表单升级为可扫描列表。
- 测试结果贴近触发点。
- 审计从长表格升级为可读 diff。
- 破坏性操作和保存影响必须在交互上更明确。

优先做 P0 项即可明显改善使用体验；P1 完成后，该模块会更适合长期维护多 Provider
和图片生成等长耗时链路。
