# 空间统计能力设计

修改时间：2026-07-01

适用范围：微信小程序首页、空间页、美食库、打卡点、菜单安排、Go 后端空间相关接口。

## 0. 修订说明（2026-07-01 体验收敛）

基于交互体验评审，对原设计做了 5 处"做减法"调整，方向不变、范围更聚焦：

1. **移除"快速安排下一餐"**：随机推荐 3 道菜不是"行动建议"而是把决策甩给概率，易触发反复刷新。整体从需求中删除，待想清楚推荐策略后再评估。
2. **指标分层收敛**：区分"用户价值指标"（默认露出）与"数据健康指标"（图片覆盖、解析率、POI 覆盖等，折叠或仅在补全引导时出现），避免半屏退化成运营仪表盘。
3. **半屏内改按钮刷新**：半屏抽屉的下拉手势保留给"关闭"，刷新改为显式"刷新"按钮 + 更新时间；下拉刷新只保留在空间页主卡片。
4. **去掉"本周/周末"伪时效文案**：当前字段不支持时间过滤，"本周可选/周末可去"名不副实。统一改为诚实的"想吃"/"想去"标签（仍指向行动，但不暗示时效）。
5. **V1 再收一刀**：V1 只做"统计卡 4 数字 + 单 Tab 总览（资产 + 高分复访 Top 3 + 待行动）+ 筛选跳转"。四 Tab 详情、列表轻量计数下沉到 V1.1，先验证用户是否会看统计。

## 1. 结论

统计能力建议定位为”空间概览 / 空间洞察”，以当前空间为唯一主维度，而不是分别做
“美食库统计”和”打卡库统计”两个割裂入口。

推荐第一阶段落地：

1. 在底部 `空间` 页的当前空间卡片下方新增”空间概览”卡片。
2. 卡片展示 4 个核心数字：菜品数、打卡点数、已安排菜单天数、想吃 + 想去。
3. 空间页主卡片支持**下拉刷新**，主动触发数据同步 + 统计重新聚合，并显示明确的同步状态。
4. 点击”查看洞察”打开半屏详情。**V1 只做”总览”单 Tab**（资产总量 + 高分复访 Top 3 + 待行动）；美食库 / 打卡库 / 菜单安排三个分组 Tab 下沉到 V1.1。
5. **总览 Tab 直接露出”高分复访推荐 Top 3”**，让用户一眼看到”下次可以去哪”。
6. 半屏内**不使用下拉刷新**（下拉手势留给关闭抽屉），改为顶部”刷新”按钮 + 更新时间。
7. V1 前端先基于已同步的 `recipes`、`places`、`mealOrderStore`、`kitchenMembers`
   本地聚合，不强依赖后端统计接口。
8. 后端已提前补齐 `GET /api/kitchens/{kitchenID}/stats?window=30d` 聚合接口，可用于趋势、
   成员贡献、历史行为和大数据量场景；前端 V1 仍可先本地聚合，后续按需切换。

不建议新增底部第四个”统计”Tab。统计属于低频回看和空间经营感增强，放在 `空间` 页
最符合现有信息架构；美食库和打卡点列表页只保留轻量计数，避免干扰”找菜 / 找店”的
主任务。

## 2. 当前依据

### 2.1 前端结构

当前首页主容器通过 `activeSection` 在 `library` 和 `kitchen` 间切换；`library`
内部再通过 `appMode` 在 `cook` 和 `explore` 间切换：

- `pages/index/index.vue`
  - `activeSection === 'library'`：承载美食库和打卡点。
  - `appMode === 'cook'`：美食库。
  - `appMode === 'explore'`：打卡点。
  - `activeSection === 'kitchen'`：空间页。
- `pages/index/components/kitchen-section.vue`
  - 已展示当前空间、空间身份、成员、邀请成员、加入邀请码和退出空间。
  - 适合承接空间级统计入口。

### 2.2 后端接口与数据归属

现有后端业务接口都围绕 `kitchenID` 组织：

| 能力 | 当前接口 | 说明 |
| --- | --- | --- |
| 空间成员 | `GET /api/kitchens/{kitchenID}/members` | 成员数与成员身份 |
| 美食库 | `GET /api/kitchens/{kitchenID}/recipes` | 菜谱列表 |
| 打卡库 | `GET /api/kitchens/{kitchenID}/places` | 打卡点列表 |
| 菜单安排 | `GET /api/kitchens/{kitchenID}/meal-plans` | 草稿和已提交菜单 |
| 智能添加 | `POST /api/kitchens/{kitchenID}/add-link-previews` | 菜谱 / 打卡地统一识别 |

因此统计口径应统一为“当前空间内的数据快照”。空间切换后，统计也随 `currentKitchenId`
切换。

## 3. 产品目标

### 3.1 用户目标

- 进入空间页后，能快速知道这个空间积累了多少菜、多少店、多少菜单。
- **能从统计中直接获得下一步行动**，例如”想吃 38 道菜可选””想去 12 家店可去”。
- 能回看这个空间近期是否在持续使用，而不是只看到静态成员列表。
- **能把”去过的店”和”吃过的菜”沉淀成快速决策**，例如”下次聚餐去哪””这家店点什么”。
- **能主动刷新统计数据**，确保看到最新的空间状态（尤其是多成员协作场景）。

### 3.2 产品目标

- 强化”共享空间”的经营感，而不是只把空间页做成设置页。
- 把美食库、打卡库、菜单安排串成一个统一资产面板。
- **让统计从”数据展示”转向”行动建议”**：优先露出高价值推荐（高分店铺、待尝试菜品），并提供一键跳转到对应列表筛选（行动 = 帮用户更快找到，而非替用户随机决定）。
- **增强空间活跃度感知**：通过轻量可视化（迷你图表、时间线）让用户看到”这个空间在持续使用”。

### 3.3 技术目标

- V1 不新增数据库迁移和后端接口，优先基于已同步数据前端聚合。
- 聚合逻辑尽量纯函数化，方便后续迁移到后端接口。
- **统计口径必须随空间切换、数据同步、本地缓存状态保持一致，并提供明确的”数据新鲜度”标识**。
- 后续新增后端统计接口时，接口响应结构与前端 V1 数据模型尽量兼容。

## 4. 信息架构

### 4.1 主入口

位置：`空间` 页，当前空间卡片下方，成员面板上方。

推荐文案：

```text
空间概览
这个空间已经沉淀了 38 道菜和 12 个打卡点

菜品      38
打卡点    12
已安排    5 天
想吃      16 道   ← 替代”待探索”，直接指向行动（不再用”本周”暗示时效）
想去      8 家

[查看洞察]
```

入口规则：

- 当前空间未连接时，隐藏统计卡或展示同步占位。
- 有缓存但同步失败时，展示”本地缓存”弱提示。
- **支持下拉刷新手势（仅空间页主卡片）**：触发 `recipe-store.js` / `place-store.js` 的同步方法，并在统计卡显示”同步中”状态；同步完成后自动重新聚合统计数据。
- 空空间时展示引导：**”添加第一道想吃的菜，开启你的美食空间”** + 直接弹出添加面板。
- 不在美食库 / 打卡点列表顶部放完整统计，以免挤压列表主任务。

### 4.2 详情层级

V1 建议使用半屏抽屉，而不是新增独立路由。原因是统计详情第一阶段信息量有限，半屏更
轻，且能保留用户对当前空间页的上下文。

**V1 只做”总览”单 Tab**：先把最高价值的资产总量 + 高分复访 Top 3 + 待行动跑通，验证
用户是否会看统计。美食库 / 打卡库 / 菜单安排三个分组 Tab 下沉到 V1.1，避免首版信息过载。

V1 详情结构（单 Tab）：

```text
空间洞察                          [刷新]   ← 显式刷新按钮，替代半屏内下拉刷新
更新时间：刚刚同步 / 2 分钟前              ← 明确数据新鲜度

总览：
- 空间资产：菜品、打卡点、菜单、成员
- **高分复访推荐 Top 3**（直接露出，配店铺图标 + 评分 + 推荐菜）
- 待行动：想吃菜、想去店、草稿菜单、缺定位打卡点
- 近期动态：近 30 天新增 / 打卡 / 安排（V1 轻量文字，V2 考虑迷你折线图或时间线）
```

> 交互说明：半屏抽屉的下拉手势保留给”关闭”，刷新统一走右上角”刷新”按钮，避免手势冲突。

V1.1 再引入分组 Tab（美食库 / 打卡库 / 菜单安排），并遵循”指标分层”原则（见 §5.5）。

V2 若加入趋势图、成员贡献和历史统计，再考虑新增：

```text
pages/space-stats/index.vue
```

## 5. 指标设计

### 5.1 空间总览指标

| 指标 | 口径 | V1 数据来源 | 展示建议 |
| --- | --- | --- | --- |
| 菜品数 | 当前空间全部未删除菜谱数量 | `recipes.length` | 大数字 |
| 打卡点数 | 当前空间全部未删除打卡点数量 | `places.length` | 大数字 |
| 已安排菜单天数 | 已提交菜单记录数量 | `mealOrderStore.submitted.length` | 大数字 |
| 成员数 | 当前空间成员数量 | `kitchenMembers.length` | 大数字 |
| 想吃 | 想吃菜数量 | `recipe.status === 'wishlist'` | **替代"待探索"，直接指向行动；不再用"本周"暗示时效** |
| 想去 | 想去打卡点数量 | `place.status === 'want'` | **替代"待探索"，直接指向行动；不再用"周末"暗示时效** |
| 高分复访推荐 Top 3 | `revisitRating >= 4` 的店铺，按评分排序 | `places.filter(...)` | **总览 Tab 直接露出**，配图标 + 评分 + 推荐菜 |

### 5.2 美食库指标

| 指标 | 口径 | 价值 |
| --- | --- | --- |
| 早餐 / 正餐数量 | 按 `mealType` 聚合 | 看菜谱结构是否偏科 |
| 想吃 / 吃过数量 | 按 `status` 聚合 | 看待尝试池和已沉淀规模 |
| 图片覆盖率 | 有 `imageUrls` 的菜谱 / 全部菜谱 | 列表质感与识别度 |
| 已解析菜谱数 | `parseStatus === 'done'` 或有结构化步骤 | 看菜谱内容完整度 |
| 可生成步骤图数 | 有结构化步骤但未生成流程图 | 引导后续补齐 |
| 最近新增菜谱 | 近 30 天 `createdAt` | 空间活跃度 |

V1 不建议做“吃过趋势”。当前菜谱只有当前 `status`，没有 `doneAt` 或状态历史，无法
准确知道是哪一天从想吃变成吃过。

### 5.3 打卡库指标

| 指标 | 口径 | 价值 |
| --- | --- | --- |
| 想去 / 去过数量 | 按 `status` 聚合 | 看待探索池和已完成规模 |
| 有定位比例 | 有 `latitude` 和 `longitude` 的地点 / 全部地点 | 判断能否直接导航 |
| 已打卡体验完整度 | 去过且有 `revisitRating` 或 `recommendedItems` | 判断记忆是否可复用 |
| 重访推荐数 | `revisitRating >= 4` | 复访候选 |
| 低分避雷数 | `revisitRating <= 2` | 避免重复踩坑 |
| 推荐项 Top | 聚合 `recommendedItems` | 回答“去这家点什么” |
| 场景标签 Top | 聚合 `scenes` 和 `tags` | 回答“聚餐 / 约会 / 顺路去哪” |
| POI 匹配覆盖 | 有 `externalProvider` 和 `externalPoiId` 的地点数量 | 判断外部数据可刷新程度 |

打卡点统计比美食库更适合先做体验洞察，因为 `visitedAt`、`revisitRating`、
`recommendedItems` 等字段已经能表达行为结果和主观体验。

### 5.4 菜单安排指标

| 指标 | 口径 | 价值 |
| --- | --- | --- |
| 已安排天数 | `submitted.length` | 空间计划使用程度 |
| 草稿天数 | `Object.keys(drafts).length` 且有菜品 | 提醒继续完成 |
| 最近安排 | 最近一个 `submitted` | 回看空间行为 |
| 下一次安排 | `planDate >= today` 的最早菜单 | 提醒即将吃什么 |
| 菜单平均菜数 | 已提交菜单 `items.length` 平均值 | 看菜单规模 |
| 菜单菜品来源 | `items.mealTypeSnapshot` 聚合 | 看早餐 / 正餐使用情况 |

### 5.5 指标分层原则

上述 5.2 / 5.3 / 5.4 指标混合了两类视角，展示时必须分层，避免半屏退化成"运营仪表盘"、
冲淡真正高价值的"高分复访 Top 3"：

- **用户价值指标（默认露出）**：直接回答"下次吃什么 / 去哪 / 点什么 / 什么时候吃"。
  例如 想吃 / 想去数量、高分复访 Top 3、低分避雷、推荐项 Top、场景标签 Top、下一次安排。
- **数据健康指标（默认折叠或仅在补全引导中出现）**：反映数据完整度、偏运营视角，用户
  不会据此决策。例如 图片覆盖率、已解析菜谱数、可生成步骤图数、有定位比例、POI 匹配覆盖。

落地建议：

- 分组 Tab（V1.1 起）内，用户价值指标优先排布在上方；数据健康指标折叠进"数据完整度"
  分区，或仅在存在缺项时以"还有 X 道菜没有图片，去补全"这类**引导型提示**出现，而不是
  常驻的百分比数字。
- 空态 / 补全引导是数据健康指标的主要用武之地——把"覆盖率低"转化为一个可点击的补全动作，
  而不是一个让用户焦虑的分母。

## 6. 页面与交互设计

### 6.1 空间概览卡

建议新增组件：

```text
pages/index/components/space-stats-card.vue
```

组件职责：

- 接收 `stats`、`isSyncing`、`hasKitchen`、`isCacheSnapshot`。
- 展示 4 个核心数字和一条行动建议。
- 点击卡片或“查看洞察”向父级发送 `open-stats`。

建议视觉：

- 延续当前空间页暖白、陶土、鼠尾草绿配色。
- 使用 2x2 数字格，数字比标签更突出。
- 避免复杂图表，优先使用进度条、横向条和轻量标签。
- 使用 `uview-plus` 图标或现有静态 SVG，不用 emoji 当 UI 图标。

### 6.2 空间洞察半屏

建议新增组件：

```text
pages/index/components/space-stats-sheet.vue
```

Tab 结构（分阶段）：

**V1（单 Tab）：**

| Tab | 内容 |
| --- | --- |
| 总览 | 资产总量、**高分复访推荐 Top 3**、待行动、近期动态 |

**V1.1 起（引入分组 Tab，遵循 §5.5 指标分层）：**

| Tab | 内容 |
| --- | --- |
| 美食库 | 餐别 / 状态结构（图片覆盖、解析完整度折叠进"数据完整度"） |
| 打卡库 | 想去 / 去过、重访评分分布、推荐项、场景标签（有定位比例折叠） |
| 菜单安排 | 草稿、已提交、下一次安排、平均菜数 |

交互规则：

- 统计数字支持点击跳转到对应列表筛选。
- “想吃”跳到美食库并设置 `appMode='cook'`、`activeStatus='wishlist'`。
- “想去”跳到打卡点并设置 `appMode='explore'`、`activePlaceStatus='want'`。
- **”高分复访推荐”点击店铺直接跳到打卡点详情**（如果有独立详情页）或筛选定位。
- “草稿菜单”跳到点菜模式或菜单详情入口。
- 缺定位地点先跳到打卡点列表，后续可支持独立筛选。
- **刷新走右上角显式"刷新"按钮**（不在半屏内做下拉刷新，下拉手势留给关闭抽屉）；刷新后自动更新各 Tab 数据，并更新"更新时间"。

### 6.3 美食库 / 打卡点列表轻量露出

列表页不放完整统计，只做局部增强：

- 美食库筛选条继续展示餐别数量和状态数量。
- 打卡点状态筛选补充 `全部 / 想去 / 去过` 数量。
- 列表标题根据筛选显示当前结果数。

## 7. 前端实现方案

### 7.1 聚合模块

建议新增纯函数模块：

```text
utils/space-stats.js
```

职责：

- `buildSpaceStats({ recipes, places, mealOrderStore, members, now })`
- 输出稳定结构供卡片和半屏复用。
- 不直接依赖 `uni`、页面实例或接口请求。

推荐数据结构：

```js
{
  updatedAt: '',
  source: 'remote' | 'cache',
  isSyncing: false,  // 新增：标识正在同步
  overview: {
    recipeTotal: 0,
    placeTotal: 0,
    submittedMealPlanDays: 0,
    memberTotal: 0,
```

## 8. 后端演进方案

### 8.1 V1 不新增接口

V1 使用前端已同步数据聚合，原因：

- 当前首页已经拉取 `recipes`、`places`、`meal-plans`、`members`。
- 统计只做当前快照和轻量分布，前端计算成本低。
- 不涉及数据库迁移，风险小。

### 8.2 V2 新增统计接口

当出现以下需求时，再新增后端接口：

- 近 7 天 / 近 30 天趋势。
- 成员贡献排行。
- 菜谱状态变化历史。
- 打卡消费统计。
- 数据量增大后前端列表不再全量加载。

建议接口：

```text
GET /api/kitchens/{kitchenID}/stats?window=30d
```

响应结构尽量与 V1 `spaceStats` 对齐。

后端模块建议：

```text
backend/internal/spacestats/
  model.go
  repository.go
  service.go
  handler.go
```

路由放在：

```go
protected.Get("/kitchens/{kitchenID}/stats", spaceStatsHandler.Overview)
```

服务层必须校验当前用户是该空间成员，口径与 `recipe`、`place`、`mealplan` 保持一致。

### 8.3 后续可能需要的数据增强

| 需求 | 当前限制 | 后续增强 |

1. **V1 只做前端闭环**：基于首页已经同步到内存和本地缓存的 `recipes`、`places`、
   `mealOrderStore`、`kitchenMembers` 聚合，前端不依赖新增后端接口或新增数据库迁移。

## 10. 数据口径与风险

### 10.1 口径约束

- 统计默认只看当前空间，不跨空间汇总。
- V1 统计是“当前快照”，不是完整历史。
- `wishlist` 表示“想吃”，`done` 表示“吃过”。
- `want` 表示“想去”，`visited` 表示“去过”。
- 打卡点外部评分 `rating` 来自高德等第三方，不能与用户重访评分 `revisitRating`
  混合计算。
- `price` 仍作为原始文本展示；后端已额外维护内部结构化金额字段，后端 stats 可做
  金额求和和平均，现有打卡点接口不暴露这些内部字段。

### 10.2 空态策略

| 场景 | 展示策略 |
| --- | --- |
| 没有空间 | 不展示统计，提示创建或加入空间 |
| 空空间 | 展示 0 值统计和**温暖的引导文案**：”添加第一道想吃的菜，开启你的美食空间” + 直接弹出添加面板 |
| 同步中 | 展示骨架态或”正在同步”，统计卡顶部显示加载指示器 |
| 同步失败但有缓存 | 展示缓存统计，标记”本地缓存 · 2 分钟前” + 提供”重新同步”入口 |
| 同步失败且无缓存 | 展示错误提示和重试入口 |

### 10.3 性能风险

V1 前端聚合复杂度为 `O(n)`，当前阶段可接受。需要注意：

- `spaceStats` computed 内避免重复多次遍历大数组，可在纯函数中一次聚合。
- 推荐项和标签 Top 聚合需要限制结果数，例如最多 5 个。
- 日期解析要兼容空字符串和非标准格式，失败时跳过。

## 14. 用户体验优化要点总结

基于产品目标和用户需求，以下是关键的体验优化方向：

### 14.1 数据时效性与透明度

- **问题**：前端聚合基于已缓存数据，多成员协作时可能陈旧。
- **方案**：下拉刷新 + 明确的"更新时间"/"同步中"状态 + "本地缓存"弱提示。

### 14.2 从数据展示到行动建议

- **问题**："待探索"概念模糊，用户不知道该去哪个列表；而"本周可选/周末可去"用了时间词，但当前字段并不支持时间过滤，属于伪时效、制造认知噪音。
- **方案**：拆分为"想吃 X 道"和"想去 X 家"，点击分别跳转到对应筛选。诚实、清晰、仍指向行动。

### 14.3 高价值内容前置

- **问题**：`revisitRating`、`recommendedItems` 是最有价值的部分，但藏在详情深处。
- **方案**：总览 Tab 直接露出"高分复访推荐 Top 3"，配图标 + 评分 + 推荐菜。

### 14.4 空间活跃度可视化

- **问题**：静态数字堆砌，缺少"持续使用"的感知。
- **方案**：V1 用近期动态文字，V2 考虑迷你折线图或时间线卡片。

### 14.5 菜单安排联动（已收敛）

- **问题**：看到"已安排 5 天"后，下一步动作不明确。
- **原方案的坑**：曾计划"快速安排下一餐"按钮从想吃池**随机**推荐 3 道菜。但美食是强主观决策，随机推荐命中率低，会诱导用户反复刷新，比手动选更累——随机不是"行动建议"。
- **当前处理**：V1 移除该功能。后续若重做，需带轻量策略（如早餐+正餐搭配、图片完整优先、避开近期已吃）或降级为"跳到想吃池并预选中"，把最终决策权还给用户，再评估上线。

### 14.6 空态引导更温暖

- **问题**：空空间只展示"添加入口"，缺少温度。
- **方案**："添加第一道想吃的菜，开启你的美食空间" + 直接弹出添加面板。

### 14.7 可视化增强（V1.1 / V2）

- 重访评分分布：轻量饼图或横向条。
- 场景标签：标签云（气泡大小表示频次）。
- 地图热力图（V2）：在"打卡库"中嵌入地图，标注"去过的店"。

## 15. 视觉设计系统与动效规范

> 本章补充统计能力的 UI 视觉、交互与动效标准。目标：与现有页面**风格统一**、具备**高级感**、
> 有**恰到好处**的动态效果，而非炫技。所有取值都尽量复用现有页面已在用的暖白木质色板，
> 避免引入第二套视觉语言。本章为设计规格，落地时再转成 SCSS / SFC，不在本轮编码。

### 15.1 设计基调：Nature Distilled（自然萃取）

统计面板的高级感**不来自冷科技仪表盘风**（深色 + 霓虹 + 玻璃拟态），而来自"暖白木质 + 手作温度"
的**自然萃取**质感——恰好契合"美食 / 生活方式内容"的产品调性，也与首页、详情页现有观感一致。

- 关键词：暖陶土、砂米、橄榄绿、奶油白、柔和阴影、极淡纸张纹理、自然缓动。
- 高级感三支柱：**克制的色彩**（强调色只服务于"行动 / 推荐"）、**柔和的分层深度**（多层软阴影而非硬边框）、**呼吸感留白**（数字周围留白充足，信息不拥挤）。
- 明确不做：玻璃拟态强模糊、彩色渐变大色块、霓虹描边、装饰性无限动画。

### 15.2 设计 Token（建议集中化）

**现状问题**：配色目前内联散落在各页面（`pages/index` 内有 26 次 `#8a7d70`、多处 `#41362d` 等），
没有集中的 design token。统计能力是新增模块，建议**借此建立一层语义化 SCSS 变量**（放在
`uni.scss` 或新增 `styles/tokens.scss`），统计组件优先引用变量，后续可反向收敛老页面。

色板 token（取值来自现有页面实测 + Nature Distilled 印证，均为暖调）：

| 语义 Token | 建议值 | 用途 |
| --- | --- | --- |
| `--bg-page` | `#fffaf3` | 页面暖白底 |
| `--bg-card` | `#ffffff` | 统计卡 / 半屏卡片面 |
| `--bg-sunken` | `#f8f4ee` | 数字格、次级分区底（下沉感） |
| `--line-soft` | `#ece2d5` | 极淡分隔线 / 卡片描边 |
| `--text-strong` | `#2f2923` | 大数字、标题 |
| `--text-base` | `#41362d` | 正文 |
| `--text-muted` | `#8a7d70` | 标签、辅助说明 |
| `--brand-terracotta` | `#bf715f` | 主强调 / 主行动（想吃、主按钮） |
| `--brand-caramel` | `#f4a236` | 高价值高亮（高分复访、评分星） |
| `--brand-sage` | `#8a9a5b` | 完成 / 沉淀态（吃过、去过） |
| `--shadow-soft` | `0 2rpx 8rpx rgba(60,42,30,.06)` | 卡片常态浮起 |
| `--shadow-float` | `0 8rpx 24rpx rgba(60,42,30,.10)` | 半屏 / 按下抬升 |

语义配色映射（**需与现有列表页筛选态用色核对，避免同一状态两种颜色**）：

| 状态 | 颜色 | 理由 |
| --- | --- | --- |
| 想吃 / 想去（待行动） | `--brand-caramel` 焦糖橙 | 明亮、召唤行动 |
| 吃过 / 去过（已沉淀） | `--brand-sage` 鼠尾草绿 | 安定、完成感 |
| 高分复访推荐 | `--brand-terracotta` + 焦糖橙星 | 最高价值，暖而不刺激 |
| 低分避雷 | `--text-muted` 灰褐（**不用红**） | 避免制造焦虑，弱化呈现 |

尺寸 token：圆角 `--radius-card: 20rpx` / `--radius-chip: 999rpx`；间距节奏用 8rpx 栅格
（`8 / 16 / 24 / 32rpx`），卡片内边距 `28~32rpx`，保证"呼吸感留白"。

### 15.3 排版：让数字成为主角

统计卡的核心是"一眼看到数字"，排版必须建立**数字 > 标签**的强层级：

| 角色 | 字号（rpx） | 字重 | 颜色 |
| --- | --- | --- | --- |
| 核心大数字（菜品 / 打卡点…） | 48~56 | 600~700 | `--text-strong` |
| 数字单位（道 / 家 / 天） | 24 | 400 | `--text-muted` |
| 指标标签 | 24~26 | 400 | `--text-muted` |
| 卡片标题（空间概览） | 30 | 600 | `--text-base` |
| 一句话行动建议 | 26 | 500 | `--text-base` |

- 小程序无法自由引入 web 字体，统一用系统字体，靠**字号 + 字重 + 颜色**三档建立层级，不依赖字体族。
- 数字建议启用等宽数字特性（`font-variant-numeric: tabular-nums`）避免跳动时宽度抖动。

### 15.4 高级感的四个具体手法

1. **柔和分层阴影**（借鉴 Soft UI Evolution，但保持 AA 对比度）：卡片用 `--shadow-soft` 常态浮起，
   按下 / 半屏用 `--shadow-float`；**用软阴影分层代替硬边框**，是廉价感与高级感的分水岭。
2. **数字格下沉**：2×2 数字格底用 `--bg-sunken` + 内描边高光（`inset 0 1rpx 0 rgba(255,255,255,.6)`），
   形成轻微凹陷的"陈列格"质感。
3. **极淡材质细节**：卡片可叠加 opacity ≈ 0.04 的纸张纹理 / grain overlay（PNG 或 SVG 噪声），
   近看有手作温度，远看仍是纯净暖白——这是 Nature Distilled 的招牌细节。
4. **强调色配额制**：一屏内强调色（陶土 / 焦糖）**只出现在"行动项"和"高分推荐"**，其余全部中性暖灰。
   色彩越克制，重点越突出，高级感越强。

### 15.5 动效规范

沿用项目现有动效基调（页面里已大量使用 `transition: transform … ease` 与
`cubic-bezier(0.2, 0.8, 0.2, 1)`），保持一致，不引入新缓动语言。

**总原则**（呼应 UX 指南）：

- **时长**：入场 240~280ms，微交互（hover / press / 状态切换）150~200ms，禁止 >500ms。
- **缓动**：入场 `ease-out`（现有 `cubic-bezier(0.2,0.8,0.2,1)`），退场 `ease-in`；UI 过渡**禁用 linear**。
- **一屏最多 1~2 个关键动效**，其余用即时或极短过渡，避免"什么都在动"的廉价感。
- 无限循环动画**仅用于 loading**，绝不用于装饰。

**具体场景动效清单：**

| 场景 | 动效 | 参数 |
| --- | --- | --- |
| 统计卡首次进场 | 卡片 `translateY(16rpx)+fade` 上浮入场 | 280ms ease-out |
| 核心大数字 | count-up 数字滚动到目标值 | 600~800ms，`ease-out`，仅首次 / 刷新后触发 |
| 高分复访 Top 3 | 三张卡 **stagger 依次**淡入上移 | 每张间隔 60ms，单张 220ms |
| 半屏抽屉 | 从底部上滑弹入 + 遮罩淡入 | 300ms，带轻微 overshoot（回弹感） |
| 顶部刷新按钮 | 点击后图标旋转 | 旋转 `animate-spin`，同步中持续，完成即停 |
| 数字刷新变化 | 旧值淡出上移 / 新值淡入 | 200ms，仅变化的数字触发 |
| 待行动项 / 卡片按下 | `scale(0.98)` + 阴影抬升，**不位移布局** | 150ms |
| 同步中 | 骨架屏 shimmer（暖白灰） | 1.2s 循环，仅 loading 期间 |

**禁忌**：hover / press 用会挤动布局的 scale 放大（改用背景 / 阴影 / 轻微内缩）；5 个以上元素同时动；
count-up 每次进入 Tab 都重放（只在首次 + 刷新后）。

### 15.6 加载 / 空态的视觉

- **同步中**：用骨架屏（`--bg-sunken` 打底 + 更浅高光 shimmer），而非纯灰 spinner，保持暖调；
  数字位用圆角占位块。
- **空空间**：线性 SVG 插画（陶土色描边、奶油填充）+ 温暖引导文案"添加第一道想吃的菜，开启你的美食空间"
  + 主按钮（`--brand-terracotta`）。**不用 emoji 当插画或图标**。
- **缓存 / 同步失败**：顶部弱提示 chip（`--bg-sunken` 底、`--text-muted` 字）"本地缓存 · X 分钟前"，
  右侧"重新同步"文字按钮，视觉弱于正常数字，不抢注意力。

### 15.7 无障碍与降级（硬性）

- **`prefers-reduced-motion: reduce`**：关闭 count-up、stagger、抽屉 overshoot、shimmer，改为
  即时呈现 / 极短淡入；这是 UX 高优先级项，必须实现。
- **对比度 ≥ 4.5:1**：`--text-base #41362d` on `--bg-page #fffaf3` 满足 AA；`--text-muted` 仅用于
  非关键辅助文字，不承载正文。
- **色彩不作唯一指示**：想吃 / 想去、高分 / 避雷除颜色外，必须叠加图标或文字标签（色盲可辨）。
- **触达区**：可点统计项、chip、按钮最小可点区域 ≥ 44×44px（约 88rpx）。
- **375px 宽度**：2×2 数字格、Top 3 卡片、chip 均需在 375px 下不溢出、不换行错位。

### 15.8 组件级视觉规格（呼应 §6 组件）

**`space-stats-card`（空间概览卡）**

- 结构：标题行（"空间概览" + 一句话资产）｜ 2×2 大数字格 ｜ 想吃 / 想去行动条 ｜"查看洞察"入口。
- 视觉：`--bg-card` + `--radius-card` + `--shadow-soft`；数字格用 `--bg-sunken` 下沉格。
- 强调：仅"想吃 / 想去"数字与"查看洞察"箭头用强调色，其余中性。

**`space-stats-sheet`（半屏总览，V1 单 Tab）**

- 顶部：居中 grabber 拖动条（`--line-soft`）｜ 左"空间洞察"标题 ｜ 右"刷新"按钮 + "更新时间"。
- 内容分区（自上而下，符合"高价值前置"）：资产总量 → **高分复访 Top 3** → 待行动 → 近期动态。
- Top 3 卡片：左店铺缩略图（圆角 `16rpx`）｜ 右店名 + 焦糖橙评分星 + 推荐菜 chip（最多 2 个）。
- 刷新走顶部按钮（§6.2），**半屏内不做下拉刷新**（下拉手势留给关闭）。

### 15.9 落地检查清单（交付前自检）

- [ ] 未使用 emoji 当 UI 图标 / 插画（统一 uview-plus 图标或静态 SVG）。
