# 菜谱详情页流程图性能改造方案

更新时间：`2026-04-25 CST`

## 1. 背景

当前小程序菜谱详情页已经支持：

- 在详情页内查看流程图缩略展示
- 进入横屏查看页查看完整流程图
- 手动触发流程图生成
- 后端按队列异步生成并回写图片

但从实际体验看，“流程图打开慢”不是单一问题，而是两类体验混在一起：

1. 已经生成好的流程图，用户再次打开详情页或进入横屏查看时，图片出现慢。
2. 用户手动点击“生成流程图”后，从提交到可查看结果的总等待时间偏长。

本方案的目标是先把这两类问题拆开，再按收益和风险分阶段优化。

## 2. 现状链路

### 2.1 前端查看链路

当前前端查看链路大致如下：

1. 详情页加载菜谱详情，拿到 `flowchartImageUrl`
2. 详情页直接用远程 `flowchartImageUrl` 渲染流程图卡片
3. 流程图图片加载成功后，立即走 canvas 裁一张正方形临时图，用于分享封面
4. 用户点击“横屏查看”时，详情页把 `flowchartImageUrl` 放进本地 storage
5. 横屏查看页读取这个 URL，再次按远程图片加载

### 2.2 后端生成链路

当前后端生成链路大致如下：

1. 前端调用 `POST /api/recipes/{recipeID}/flowchart`
2. 接口只负责把当前菜谱标记为 `pending`
3. worker 按固定间隔轮询待处理任务
4. worker 调模型生成流程图图片
5. 后端再把模型返回的远程图片下载到本地 uploads
6. 菜谱详情接口返回新的 `flowchartImageUrl`

## 3. 已确认的主要瓶颈

### 3.1 已生成流程图的查看慢

这是当前最值得优先解决的问题，原因明确，改造风险也最低。

#### 瓶颈 A：流程图没有复用现有本地图片缓存能力

首页和详情页普通成品图已经接入 `utils/image-cache.js`，会优先命中本地缓存文件，再兜底远程 URL。

但流程图查看链路没有接入这套缓存：

- 详情页流程图直接使用远程 `flowchartImageUrl`
- 横屏查看页也直接再次加载远程 `imageUrl`

结果是：

- 同一个用户刚在详情页里看到流程图，进入横屏查看时仍可能再次拉取大图
- 再次进入详情页或横屏查看时，也不能稳定命中本地已缓存文件
- 在弱网或图片较大时，“已生成但打开慢”的主观感受会非常明显

#### 瓶颈 B：分享封面裁图放在首屏图片加载阶段

当前详情页里，流程图图片一旦加载成功，就会立刻用 canvas 裁一张正方形临时图作为分享封面候选。

这条逻辑本身没错，但它的触发时机不理想：

- 用户只是想看流程图时，也会触发裁图
- 裁图会占用额外主线程、canvas 和文件 I/O
- 在真机上，这部分额外开销会和“图片首屏出现”叠在一起

这意味着分享能力的准备动作，提前阻塞了查看体验。

### 3.2 手动生成后的总等待长

这部分不是单点问题，主要是异步链路的固定延迟叠加。

#### 瓶颈 C：worker 采用固定间隔轮询，存在 pickup delay

当前流程图生成请求提交后，接口本身不会立即执行生成，而是先标记成 `pending`，再等 worker 下一轮扫描。

这会带来一个天然等待窗口：

- 用户刚点击“生成”，任务不一定立刻被 worker 捞起
- 默认 `interval=5s`、`batchSize=1` 时，排队感知会偏明显
- 即使没有前置排队，也可能先空等几秒才真正开始生成

#### 瓶颈 D：后端保存的是原始下载图，没有展示版衍生图

当前后端拿到模型返回图片后，会把远程图片原样下载并写入 uploads。

如果模型返回的是体积偏大的 `png` 或高分辨率图片，那么：

- 首次查看下载慢
- 再次查看解码也慢
- 小程序端在横屏查看时更容易放大这个问题

#### 瓶颈 E：状态轮询走完整菜谱详情

当流程图处于 `pending / processing` 时，前端会周期性拉取完整菜谱详情。

这样做实现简单，但不是最经济的方式：

- 只关心状态变化时，仍然要返回整份菜谱详情
- 会增加接口负担，也会增加前端响应式更新成本

## 4. 改造目标

本次改造先以“改善用户体感”作为第一目标，不追求一次性把整条链路全部重写。

### 4.1 用户体验目标

1. 对于已经生成完成的流程图，再次打开详情页时应明显快于当前版本。
2. 从详情页进入横屏查看时，不应再像首次远程下载大图一样明显等待。
3. 分享封面准备逻辑不应阻塞流程图首屏展示。
4. 用户点击“生成流程图”后，应尽快看到明确的排队/处理中反馈，且等待时间口径更可信。

### 4.2 工程目标

1. 优先复用现有图片缓存能力和现有页面结构，避免大范围重构。
2. 将“查看链路优化”和“生成链路优化”分阶段实施，控制风险。
3. 为后续加入展示版图片和轻量状态接口留出明确扩展点。

## 5. 分阶段改造方案

## 5.1 P0：前端查看链路优化

这是第一阶段，也是建议最先落地的阶段。

### 方案 1：流程图接入本地图片缓存

实施要点：

1. 详情页在拿到 `flowchartImageUrl` 后，复用 `utils/image-cache.js` 做缓存预热。
2. 详情页展示流程图时优先使用本地缓存路径，远程 URL 仅作为兜底。
3. 横屏查看页接收 `localPath + remoteURL` 两套数据，优先渲染本地路径。
4. 缓存 key 需要带版本信息，避免旧图覆盖新图。

推荐版本口径：

- 优先使用 `flowchartUpdatedAt`
- 如后续后端补了 `flowchartSourceHash` 或展示版 URL，也可以继续升级为更稳定的版本键

预期收益：

- 二次打开详情页更快
- 从详情页进入横屏查看明显更快
- 弱网环境下体验改善最明显

### 方案 2：分享封面裁图改为懒执行

实施要点：

1. 不再在流程图 `@load` 后立即裁图。
2. 改为真正触发分享时再生成正方形临时图。
3. 如需兼顾分享首帧速度，可在页面空闲后再异步准备，而不是阻塞首屏查看。

预期收益：

- 流程图首屏展示更干净
- 降低首屏期间的 canvas 和文件 I/O 干扰

### P0 验收口径

1. 已生成流程图的菜谱，第二次进入详情页时，流程图应明显快于当前版本。
2. 从详情页点击“横屏查看”时，若本地图已存在，应接近瞬时展示。
3. 不点击分享时，不应再触发正方形裁图逻辑。
4. 分享功能仍然可用，封面图回退策略不受破坏。

## 5.2 P1：后端生成链路优化

这一阶段处理“生成慢”的核心结构问题。

### 方案 3：增加流程图展示版图片

实施要点：

1. 后端保存流程图时，同时生成适合小程序展示的衍生图。
2. 展示版建议控制长边，例如 `1600` 到 `1920` 像素。
3. 展示版优先考虑 `webp` 或更轻量的 `jpeg`，原图可继续保留为下载或后续扩展用途。
4. 前端详情页和横屏查看默认走展示版 URL。

预期收益：

- 首次查看下载更快
- 图片解码和渲染成本更低
- 弱网与低端机体验更稳

### 方案 4：增加轻量状态接口

实施要点：

1. 新增只返回流程图相关状态的接口。
2. 字段只包含：
   - `flowchartStatus`
   - `flowchartQueueAhead`
   - `flowchartEstimatedWaitSeconds`
   - `flowchartImageUrl`
   - `flowchartUpdatedAt`
3. 前端在 `pending / processing` 期间优先轮询轻量接口，而不是完整菜谱详情。

预期收益：

- 降低轮询开销
- 降低前端无关字段反复更新的成本

### P1 验收口径

1. 用户手动点击“生成流程图”后，任务进入处理状态的体感等待应明显缩短。
2. 生成完成后的首次展示速度应优于当前原图直出方案。
3. 状态轮询期间接口负载下降，前端无明显回归。

## 5.3 P2：进一步优化项

这一阶段不作为首批必须项，但建议保留。

### 可选方向

1. 根据模型或 Provider 差异，对不同场景动态选择输出格式。
2. 对大尺寸流程图增加服务端压缩质量策略。
3. 在流程图生成完成后，主动触发一次前端缓存预热。
4. 在后台可观测性中补充流程图图片大小、生成耗时、下载耗时等指标。

## 6. 实施顺序建议

建议顺序如下：

1. 先做 `P0`：
   目标是尽快解决“已生成流程图打开还是慢”的问题。
2. 再做 `P1`：
   目标是降低“手动生成后总等待时间”。
3. 最后视实际收益决定是否继续做 `P2`。

原因：

- `P0` 主要集中在前端局部逻辑，收益直接，回归风险最低。
- `P1` 会涉及接口、上传链路和状态接口改造，需要在 `P0` 稳定后再推进。

## 7. 风险与回滚

### 7.1 前端缓存相关风险

风险点：

- 缓存键不稳定导致旧图复用
- 本地缓存文件失效或被系统清理
- 详情页与横屏查看页缓存口径不一致

控制方式：

- 缓存键带版本字段
- 本地路径失效时自动回退远程 URL
- 详情页和 viewer 统一复用同一套缓存工具

回滚方式：

- 保留远程 URL 渲染兜底
- 若缓存链路异常，可单独关闭流程图本地缓存逻辑，不影响主流程

### 7.2 后端衍生图相关风险

风险点：

- 衍生图生成增加 CPU 和磁盘占用
- 原图与展示图版本不一致
- 老数据迁移期间存在兼容期

控制方式：

- 第一版允许只对新生成流程图产出展示版
- 前端按“展示版优先，原图兜底”兼容

回滚方式：

- 后端可暂时停止生成展示版
- 前端继续使用原始 `flowchartImageUrl`

## 8. 验证方案

### 8.1 前端验证

建议至少覆盖以下场景：

1. 已生成流程图的菜谱，首次进入详情页。
2. 已生成流程图的菜谱，第二次进入详情页。
3. 详情页进入横屏查看。
4. 分享前未触发裁图，分享时能正常拿到封面。
5. 弱网或低端机下流程图加载体验是否明显改善。

### 8.2 后端验证

建议至少覆盖以下场景：

1. 手动提交生成后，任务是否更快进入处理状态。
2. 生成完成后，详情页能否正确拿到展示图或原图。
3. 轮询状态接口与完整详情接口口径是否一致。
4. worker 唤醒失败时，定时扫描兜底是否仍然有效。

## 9. 非目标

本方案当前不包含以下内容：

1. 流程图视觉风格或 prompt 规则改造
2. AI Provider 路由策略改造
3. 详情页整体 UI 重构
4. 分享能力的产品策略调整
5. 将流程图 worker 从“轮询捞任务”改成“入队即唤醒”

当前暂不纳入该项，原因是：

- 流程图生成本身耗时和成本都偏高
- 当前阶段更优先解决“已生成流程图打开慢”的查看体验问题
- 在没有补齐展示版图片、轻量状态接口和缓存口径前，先做 worker 即时唤醒的收益不够集中

## 10. 建议结论

本次改造建议先把“查看慢”当成第一优先级问题处理。

最值得优先落地的两项是：

1. 流程图接入本地图片缓存
2. 分享封面裁图改为懒执行

这两项都能直接改善当前用户最容易感知到的卡顿，且实现范围集中、回滚简单。等 `P0` 稳定后，再推进展示版图片和轻量状态接口。
